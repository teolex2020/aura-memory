### CR-02: PolicyEngine Surface.ts local type definitions are structurally incompatible with contract PolicyHint (CRITICAL)

**File:** `packages/policy/src/Surface.ts:39-65`
**Previous review context:** WR-12 in the original review, classified as WARNING and SKIPPED by the fixer. This follow-up review elevates to CRITICAL because the incompatibility makes the surface layer non-functional with actual engine output.
**Issue:** `packages/policy/src/Surface.ts` defines local `PolicyHint` and `PolicyEngine` interfaces (lines 38-65) that are fundamentally incompatible with the contract `PolicyHint` type in `packages/contract/src/policy/PolicyTypes.ts`:

| Local Surface `PolicyHint` (lines 39-57) | Contract `PolicyHint` (PolicyTypes.ts:48-89) | Mismatch |
|---|---|---|
| `key: string` | `cause_key: string` | Different field name |
| `triggerCausalIds` | No such field | Only in local type |
| `triggerConceptIds` | No such field | Only in local type |
| `triggerBeliefIds` | No such field | Only in local type |
| `supportingRecordIds` | No such field | Only in local type |
| `causeRecordIds` | `cause_record_ids` | camelCase vs snake_case |
| `lastUpdated` | `last_updated` | camelCase vs snake_case |
| No `pattern_id` field | `pattern_id: string \| null` | Missing from local |
| No `polarity` field | `polarity: Polarity` | Missing from local |
| No `condition` field | `condition: string` | Missing from local |
| No `action` field | `action: string` | Missing from local |
| No `priority` field | `priority: number` | Missing from local |
| No `effect_keys` field | `effect_keys: ReadonlyArray<string>` | Missing from local |
| No `cause_key` field | `cause_key: string` | Missing from local |

The `surfacePolicyHints` and `surfacePolicyHintsFiltered` functions at lines 122-202 iterate over `engine.hints.values()` expecting the local `PolicyHint` type. Since these functions are called with engine state that contains contract-typed `PolicyHint` objects (snake_case fields, `cause_key` not `key`, `cause_record_ids` not `causeRecordIds`), the surface functions will silently fail to read the expected fields. `isHintEligible` checks `hint.triggerCausalIds.length` (line 74) which will be `undefined.length` on contract-typed hints, and `hint.domain` (line 77) which exists on both. The result is that ALL contract-typed policy hints will fail eligibility checks, and the surface will return an empty array.

This is NOT just a cosmetic naming mismatch -- it is a runtime correctness issue.

**Fix:** The surface layer must operate on the contract `PolicyHint` type directly. Either:
1. Delete the local `PolicyHint` and `PolicyEngine` interfaces and rewrite the surface functions to use `import("@aura/contract").PolicyHint` and `import("@aura/contract").PolicyEngine.Interface`, OR
2. Create a mapping/converter layer from contract types to surface-view types, and ensure all field accesses match the contract's actual field names.

Minimal immediate fix for `surfacePolicyHintsFiltered`:
```typescript
import { type PolicyHint, PolicyState, PolicyActionKind, type SurfacedPolicyHint } from "@aura/contract"
import type { PolicyEngine } from "@aura/contract"

// Remove local PolicyHint interface entirely
// Remove local PolicyEngine interface entirely

function isHintEligible(hint: PolicyHint, namespace?: string): boolean {
  if (namespace !== undefined && hint.namespace !== namespace) return false
  // Must have provenance: use contract fields
  if (hint.cause_record_ids.length === 0 || hint.effect_keys.length === 0) return false
  if (hint.domain.length === 0 || hint.recommendation.length === 0) return false
  switch (hint.state) {
    case PolicyState.Stable: return true
    case PolicyState.Candidate:
      return hint.policyStrength >= STRONG_CANDIDATE_THRESHOLD
        && hint.confidence >= MIN_SURFACE_CONFIDENCE
    case PolicyState.Suppressed:
    case PolicyState.Rejected:
      return false
  }
}

// In sort function:
// Replace a.key with a.cause_key
// Replace hint.triggerCausalIds with hint.cause_record_ids
// Replace hint.lastUpdated with hint.last_updated
```

### CR-03: PolicyEngine Surface.ts isHintEligible uses raw string comparisons instead of PolicyState enum (CRITICAL)

**File:** `packages/policy/src/Surface.ts:80-91, 166-167`
**Issue:** The `PolicyState` enum is imported at line 11 (`import { PolicyState } from "@aura/contract"`) and used in the `stateString` helper at line 31. // Sort function:
if (a.state === PolicyState.Stable && b.state !== PolicyState.Stable) return -1
if (a.state !== PolicyState.Stable && b.state === PolicyState.Stable) return 1

### CR-04: Unsafe `CognitiveRecord as unknown as AuraRecord` cast — missing required fields (CRITICAL)

**File:** `packages/core/src/Aura.ts:411`
**Issue:** At line 411:
```typescript
const records = new Map<string, AuraRecord>(
  recordsList.map((r) => [r.id, r as unknown as AuraRecord])
)
```

`r` is `CognitiveRecord` which has only 5 known fields:
```typescript
// CognitiveRecord.ts:6-12
export type CognitiveRecord = {
  id: string; content: string; tags: string[]
  aura_id?: string | null
  [k: string]: unknown  // index signature
}
```

`AuraRecord` (contract `Record`, Record.ts:5-28) has **22 fields** including required (non-optional) ones:
- `level: Level` — no default
- `strength: number` — no default
- `activation_count: number` — no default
- `created_at: number` — no default
- `last_activated: number` — no default
- `namespace: string` — no default
- `semantic_type: string` — no default
- ... plus 8 more required fields

The `as unknown as` double assertion bypasses TypeScript entirely, producing "phantom" `AuraRecord` objects where required fields are `undefined` at runtime. Downstream engines accessing `rec.namespace`, `rec.semantic_type`, `rec.level` etc. will get `undefined`, causing silent incorrect behavior in belief clustering, causal edge extraction, and policy seed gating.

The `CognitiveRecord` index signature does NOT guarantee these typed fields exist — it only guarantees that any dynamic property access returns `unknown`.

**Fix:** Create a proper conversion function with explicit defaults for all required fields:
```typescript
function cognitiveToAuraRecord(rec: CognitiveRecord): AuraRecord {
  return {
    id: rec.id,
    content: typeof rec.content === "string" ? rec.content : "",
    level: (typeof rec.level === "number" ? rec.level : 1) as Level,
    strength: typeof rec.strength === "number" ? rec.strength : 1.0,
    activation_count: typeof rec.activation_count === "number" ? rec.activation_count : 0,
    created_at: typeof rec.created_at === "number" ? rec.created_at : 0,
    last_activated: typeof rec.last_activated === "number" ? rec.last_activated : 0,
    tags: Array.isArray(rec.tags) ? rec.tags.filter((t): t is string => typeof t === "string") : [],
    connections: (rec.connections && typeof rec.connections === "object") ? rec.connections as { readonly [k: string]: number } : {},
    connection_types: (rec.connection_types && typeof rec.connection_types === "object") ? rec.connection_types as { readonly [k: string]: string } : {},
    content_type: typeof rec.content_type === "string" ? rec.content_type : "text",
    source_type: typeof rec.source_type === "string" ? rec.source_type : "unknown",
    namespace: typeof rec.namespace === "string" ? rec.namespace : "default",
    semantic_type: typeof rec.semantic_type === "string" ? rec.semantic_type : "observation",
    metadata: (rec.metadata && typeof rec.metadata === "object") ? rec.metadata as { readonly [k: string]: string } : {},
  }
}
```

**Contrast with existing code:** `CognitiveRecord.ts:20-48` already has `normalizeCognitiveRecord()` which properly validates and defaults fields. But that produces `CognitiveRecord`, not `AuraRecord`. The Aura.ts call site skips this safety net entirely with a blind cast.

### CR-05: `CognitiveRecord` is a duplicate of contract `Record` — type fragmentation from AI refactoring (CRITICAL)

**Files:** `packages/storage/src/CognitiveRecord.ts:6-12` and `packages/contract/src/record/Record.ts:5-28`
**Concern from:** User architectural review — Rust 源码对照
**Issue:** The Rust source (`aura_sdk::record::Record`) defines a **single** `Record` struct with all 22 fields. The TypeScript codebase has two distinct types representing the same entity:

| Aspect | Contract `Record` (AuraRecord) | `CognitiveRecord` |
|--------|-------------------------------|-------------------|
| Location | `packages/contract/src/record/Record.ts` | `packages/storage/src/CognitiveRecord.ts` |
| Fields | 22 typed fields | 4 fields + `[k: string]: unknown` |
| Consumers | All 4 engines + MaintenanceService | RecallView, Aura.ts (bridge) |
| Rust parity | Matches Rust `Record` struct | Minimal subset, no parity |

This type duplication causes concrete problems:
1. **Unsafe cast at Aura.ts:411** — `r as unknown as AuraRecord` bridges the two types, bypassing all type safety (CR-04)
2. **Inconsistent APIs** — `loadCognitiveRecords()` returns `CognitiveRecord`, but all engines consume `AuraRecord`
3. **Incomplete validation** — `normalizeCognitiveRecord()` only validates 4 fields, leaving 18+ fields to be filled by index signature access
4. **RecallView** — uses `CognitiveRecord` for `buildTagIndex()` and `buildAuraIndex()`, but these functions only access `id`, `content`, `tags`, and `aura_id` — all present on contract `Record`

The storage package already depends on `@aura/contract` (for `FileRead`, `FileReadError`, etc.), so there is no circular dependency barrier to using contract `Record`.

**Fix:** Eliminate `CognitiveRecord` type and consolidate on contract `Record`:

1. Change `normalizeCognitiveRecord` return type to `AuraRecord`, adding explicit defaults for all required fields:
```typescript
import type { Record as AuraRecord } from "@aura/contract"

export function normalizeCognitiveRecord(raw: unknown): AuraRecord | undefined {
  if (!raw || typeof raw !== "object") return undefined
  const o = raw as Record<string, unknown>
  const id = typeof o.id === "string" ? o.id : undefined
  if (!id) return undefined

  return {
    id,
    content: typeof o.content === "string" ? o.content : "",
    level: (typeof o.level === "number" ? o.level : 1) as Level,
    strength: typeof o.strength === "number" ? o.strength : 1.0,
    activation_count: typeof o.activation_count === "number" ? o.activation_count : 0,
    created_at: typeof o.created_at === "number" ? o.created_at : 0,
    last_activated: typeof o.last_activated === "number" ? o.last_activated : 0,
    tags: Array.isArray(o.tags) ? o.tags.filter((t): t is string => typeof t === "string") : [],
    connections: (o.connections && typeof o.connections === "object") ? o.connections as { readonly [k: string]: number } : {},
    connection_types: (o.connection_types && typeof o.connection_types === "object") ? o.connection_types as { readonly [k: string]: string } : {},
    content_type: typeof o.content_type === "string" ? o.content_type : "text",
    source_type: typeof o.source_type === "string" ? o.source_type : "unknown",
    namespace: typeof o.namespace === "string" ? o.namespace : "default",
    semantic_type: typeof o.semantic_type === "string" ? o.semantic_type : "observation",
    metadata: (o.metadata && typeof o.metadata === "object") ? o.metadata as { readonly [k: string]: string } : {},
    // optional fields
    ...(o.aura_id !== undefined && o.aura_id !== null ? { aura_id: o.aura_id as string | null } : {}),
    ...(typeof o.caused_by_id === "string" ? { caused_by_id: o.caused_by_id } : {}),
    ...(typeof o.confidence === "number" ? { confidence: o.confidence } : {}),
    ...(typeof o.support_mass === "number" ? { support_mass: o.support_mass } : {}),
    ...(typeof o.conflict_mass === "number" ? { conflict_mass: o.conflict_mass } : {}),
  }
}
```

2. Change `loadCognitiveRecords` return type to `Map<string, AuraRecord>`
3. Change `RecallView` functions to use `AuraRecord` (they only access `id`, `content`, `tags`, `aura_id` — all present)
4. Remove the `as unknown as AuraRecord` cast at `Aura.ts:411`
5. Remove `export type CognitiveRecord` and deprecate the import
6. `CognitiveRecord.ts` can be renamed to `RecordLoader.ts` or similar, reflecting its actual purpose (loading/parsing, not defining a new type)

This consolidates ~60 lines of duplicated type definition and eliminates the entire class of bugs from the `CognitiveRecord` → `AuraRecord` boundary.

### IN-07: Five separate Union-Find implementations across BeliefEngine.ts and ConceptEngine.ts

**File:** `packages/belief/src/BeliefEngine.ts` (at least 4 copies: lines 463-509, 527-588, 598-657, 668-740) and `packages/concept/src/ConceptEngine.ts` (2 copies: lines 1098-1179, 1218-1300)
**Previous review context:** IN-03 in the original review. Still present.
**Issue:** The Union-Find (Disjoint Set) pattern with `parent` array, `find` with path compression, and `union` by root is copy-pasted across 6 functions. This is ~120 lines of duplicated logic. A bug fix in any copy requires fixing all 6.
**Fix:** Extract a shared `UnionFind` utility class:
```typescript
class UnionFind {
  private parent: number[]
  constructor(n: number) { this.parent = Array.from({ length: n }, (_, i) => i) }
  find(x: number): number {
    let v = x
    while (this.parent[v] !== v) v = this.parent[v]!
    let p = x
    while (this.parent[p] !== p) {
      const next = this.parent[p]!
      this.parent[p] = v
      p = next
    }
    return v
  }
  union(a: number, b: number): void {
    const ra = this.find(a)
    const rb = this.find(b)
    if (ra !== rb) this.parent[rb] = ra
  }
}
```

### IN-10: MaintenanceService uses `as unknown as _PhaseTimings` / `_MaintenanceHotspots` pattern 9 times (INFO)

**File:** `packages/core/src/MaintenanceService.ts:413-414, 487-488, 618-619, 707-708, 763`
**Concern from:** User review — `as unknown as` 双重断言滥用
**Issue:** The helper types `_PhaseTimings` and `_MaintenanceHotspots` (lines 74-75) strip `readonly` from contract types to allow in-place mutation:
```typescript
type _PhaseTimings = { -readonly [K in keyof PhaseTimings]: PhaseTimings[K] }
type _MaintenanceHotspots = { -readonly [K in keyof MaintenanceHotspots]: MaintenanceHotspots[K] }

// Used 9 times across 5 functions:

```

The `as unknown as` chain is technically necessary here (TS forbids direct casts between types differing only by `readonly`). However, the pattern of mutating function parameters contradicts Effect's immutable-by-convention style. The 9 repetitions suggest the design should be reconsidered: either (a) construct these objects as mutable from the start, or (b) return new objects instead of mutating parameters.

**Fix:** Consider refactoring to return new objects rather than mutating. At minimum, consolidate into a single helper:
```typescript
function toMutable<T>(readonly: T): { -readonly [K in keyof T]: T[K] } {
  return readonly as unknown as { -readonly [K in keyof T]: T[K] }
}
// Single `as unknown as`, used everywhere:
const mt = toMutable(timings)
```
