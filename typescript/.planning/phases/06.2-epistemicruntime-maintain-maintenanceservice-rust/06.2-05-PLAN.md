    **Feature Group A: Complex Aggregate Methods (3)**

    A1. getBeliefInstabilitySummary():
      - Load all beliefs via engine.stats().beliefs
      - Count by state: resolved, unresolved, singleton, empty (use BeliefState enum values)
      - Compute avgVolatility: sum(volatility) / count or 0 if empty
      - Compute avgStability: sum(stability) / count or 0 if empty
      - Count highVolatilityCount: beliefs where volatility >= 0.20
      - Count lowStabilityCount: beliefs where stability <= 1.0 (default threshold)
      - Build volatilityBands:
        low: volatility < 0.20
        medium: 0.20 <= volatility < 0.50
        high: volatility >= 0.50
      - contradictionClusterCount is 0 in this method (computed by getContradictionClusters separately — leave as 0 for now, or compute from belief IDs)
      - Return BeliefInstabilitySummary

    A2. getContradictionClusters(records, namespace?, limit?):
      - Load all beliefs via engine.stats().beliefs
      - For each belief, collect record IDs (from engine state)
      - Build graph: two beliefs are connected if they share a record ID
      - Use a simple Union-Find or adjacency BFS to find connected components
      - For each cluster:
        - Collect all belief IDs in the cluster
        - Collect all record IDs across all beliefs in the cluster
        - Find shared tags (intersection of tags across beliefs — use belief_keys for now since Belief type doesn't have tags)
        - Compute avgVolatility, avgStability, totalConflictMass (sum of conflict_mass), maxConflictMass (max of conflict_mass)
        - Count unresolvedBeliefCount (state === "Unresolved")
        - Count highVolatilityBeliefCount (volatility >= 0.20)
        - Generate cluster ID from first belief ID
      - If namespace: filter clusters that contain beliefs in that namespace
      - Sort by totalConflictMass descending
      - Limit: default 20, max 100
      - Return ReadonlyArray&lt;ContradictionCluster&gt;

      Note: The `records` parameter is passed in so the method can look up record tags for shared tag computation. If records is not available, use beliefKeys for sharedTags instead. The Rust version holds a records reference — the TS version accepts it as a parameter (per RESEARCH.md Open Question #3 resolution).

    A3. getPolicyLifecycleSummary(actionLimit?, domainLimit?):
      - Load all hints via engine.stats().hints
      - Group by actionKind → PolicyActionSummary (count states, compute avgPolicyStrength, avgRiskScore)
      - Group by namespace+domain → PolicyDomainSummary (count active/stable/candidate/suppressed/rejected, compute avgPolicyStrength, avgRiskScore, advisoryPressure = activeHints / max(1, totalHints))
      - Sort actionSummaries by totalHints desc, limit by actionLimit (default 10, max 50)
      - Sort domainSummaries by totalHints desc, limit by domainLimit (default 10, max 50)
      - Compute overall totals and averages
      - Return PolicyLifecycleSummary

    A4. getPolicyPressureReport(namespace?, limit?):
      - Group hints by (namespace, domain) pair
      - For each group: compute advisoryPressure (activeHints / max(1, totalHints)), find strongest hint (highest policyStrength)
      - Sort by advisoryPressure desc
      - If namespace: filter group by namespace
      - Limit default 20, max 100
      - Return ReadonlyArray&lt;PolicyPressureArea&gt;
