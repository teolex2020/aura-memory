# Phase 06.3: Engine Algorithm Parity - Context

**Gathered:** 2026-05-28
**Status:** Ready for planning

<domain>
## Phase Boundary

将 TS 四大引擎（BeliefEngine / ConceptEngine / CausalEngine / PolicyEngine）的内部算法从"能跑通主链路的骨架实现"升级为"与 Rust 确定性一致"，修复 AUDIT-DIFF.md 中记录的 14 项系统性偏差（阈值体系、聚类策略、信号源、评分公式、守卫条件、数据结构）。

**范围覆盖：**
- BeliefEngine: 8 项偏差（P0: SDR阈值+hypothesis分裂+增量更新 / P1: 5种coarse key守卫 / P2: BridgeKey normalize+TagFamily系列）
- ConceptEngine: 4 项偏差（P1: CanonicalFeature+cluster guards+术语提取 / P2: surface函数对齐）
- CausalEngine: 5 项偏差（P0: record-level edge提取 / P1: evidence gates+评分体系 / P2: corpus fingerprint）
- PolicyEngine: 7 项偏差（P1: 三引擎输入+polarity+action mapping / P2: suppression+recommendation文本+surface对齐）

**不在本 Phase：**
- 不改变 MaintenanceService 编排逻辑（已在 Phase 06.2 对齐）
- 不改变 EpistemicRuntime inspection 方法（已在 Phase 06.2 对齐）
- 不新增引擎能力，只对齐已有方法的算法语义
</domain>

<decisions>
## Implementation Decisions

### 引擎对齐顺序与粒度
- **D-01:** 逐引擎完整对齐：Belief 全部(P0+P1+P2) → Concept 全部 → Causal 全部 → Policy 全部。每个引擎独立验证稳定后再进入下一个
- **D-02:** Belief 引擎内部按 Rust `belief.rs` 源码函数顺序从上到下逐函数翻译，保持与 Rust 相同的代码阅读顺序

### 对齐深度与实现策略
- **D-03:** 逐行 1:1 翻译 Rust 源码（方法签名 + 计算逻辑 + 常量值 + 分支条件完全一致），TS 代码结构与 Rust 逐函数对应
- **D-04:** 混合实现策略：P0 架构级变更（Belief hypothesis contradiction分裂 + Causal record-level edge提取）从 Rust 源码重写；P1/P2 参数/守卫/公式变更以 AUDIT-DIFF.md 为 checklist 逐项修复
- **D-05:** 直接读取 Rust 源码（`../src/belief.rs`、`../src/concept.rs`、`../src/causal.rs`、`../src/policy.rs`）作为翻译蓝本，EXPLAIN.md 和 AUDIT-DIFF.md 作为辅助参考

### BeliefEngine 决策
- **D-06:** Hypothesis 生成改为 Rust 的 contradiction 分裂方式（同一 coarse group 内按 contradiction 分为 supporting/opposing 两组，最多 2 个 hypotheses），替换现有 Union-Find SDR 聚类架构
- **D-07:** Hypothesis ID 从 `id12()` 随机生成改为确定性 `xxh3_64(belief_id + "\0" + sorted_record_ids)` 哈希
- **D-08:** 实现全部 5 种 SDR 子簇守卫策略（sdr_subcluster / tag_guarded / bridge_guarded / tag_sdr_guarded），对齐 Rust `coarse_key_mode` 分支
- **D-09:** SDR Tanimoto 阈值从 0.6 改为 0.15，Recency 半衰期从 7 天改为 14 天
- **D-10:** 更新模式从全量重建改为增量更新（保留 key_index / record_index，维护 churn 统计）
- **D-11:** BridgeKey normalize 桥接表作为可配置项（默认提供与 Rust 相同的映射表：ui/frontend→frontend, auth/authentication→authentication, deploy/release→release 等），支持外部覆盖
- **D-12:** TagFamilyAdaptive / TagFamilyBackoff / TagFamilyDenseBackoff 三个回退策略全部实现

### ConceptEngine 决策
- **D-13:** CanonicalFeature 模式完整实现：stemming（15 种后缀规则）+ equivalence dictionary（80+ 领域术语映射）+ Jaccard 相似度替代 SDR Tanimoto 作为默认模式
- **D-14:** Cluster guards 完整实现：tag barrier（shared_tags >= 1）+ family guard + semantic type bridge + generic family 特殊处理（"alerts" 需 shared >= 2）
- **D-15:** Surface 函数完整对齐 Rust：per-namespace cap (MAX_SURFACED_PER_NAMESPACE = 5) + dedup by key + 5 维 tiebreak + candidate 阈值 (0.70)
- **D-16:** 术语提取对齐：分词改为 `split_whitespace()` + `trim_matches(!is_alphanumeric)` + 停用词从 60+ 扩展到 80+

### CausalEngine 决策
- **D-17:** 信号源重写为 Rust 的 record-level edge 提取，实现全部三种边来源：explicit `caused_by_id` + explicit `connection_type=="causal"` + temporal edges（同 namespace 内按 `created_at` 排序，7 天窗口，cause 先于 effect）
- **D-18:** NearbySuccessors budgeting（每 cause 最多 16 个 successors）+ 每 namespace 最多 5000 条边 capping
- **D-19:** CausalPattern 评分体系完整对齐 Rust 20+ 字段（transition_lift, temporal_consistency, outcome_stability, causal_strength 等）+ evidence gates（support gate >= 2, repeated evidence gate >= 2 windows, counterfactual gate <= 0.50）
- **D-20:** 聚合层从 concept-level 改为 belief-level（cause_belief → effect_belief），使用 `record_index → hypothesis → belief` 反向索引

### PolicyEngine 决策
- **D-21:** `discover()` 扩展为接收三个引擎（causal_engine + concept_engine + belief_engine）+ records，对齐 Rust 的三引擎输入
- **D-22:** Polarity 分类完整实现（6 维度检查：semantic_type=="contradiction" + NEGATIVE_KEYWORDS + POSITIVE_KEYWORDS + content 关键词匹配），5 种 action mapping（Avoid / VerifyFirst / Prefer / Recommend / Warn）
- **D-23:** Policy strength 4 维加权公式完整对齐：`0.35*causal_strength + 0.25*confidence + 0.20*utility_score + 0.20*stability`
- **D-24:** Suppression 阶段完整实现：同 namespace+domain 冲突检测 + 正负极性冲突 + cause_record_ids 重叠 → 抑制 policy_strength 较低者
- **D-25:** Recommendation 文本逐模板对齐 Rust 的 5 种模板（Avoid / VerifyFirst / Prefer / Recommend / Warn），模板变量使用 `cause_key` 和 `domain`

### 验证与测试
- **D-26:** 四个引擎全部修完后统一跑 Rust fixture 端到端验证（非逐引擎验证）
- **D-27:** 测试以 Rust fixture parity 为主要策略（Rust fixture generator → 已知输出 → TS 断言精确匹配），保留并改写已有 TS 单元测试，新增边界条件测试

### Claude's Discretion
- Rust 源码中具体常量的提取和映射（桥接表、关键词表、停用词表、等价词典表的具体条目）
- 每个引擎内部函数的 TS 翻译细节（在签名和算法已锁定的前提下）
- 测试用例的边界条件设计
- Phase 内 plan 的拆分粒度和 wave 编排
</decisions>

<specifics>
## Specific Ideas

1. **Belief 重写路径：** 从 Rust `belief.rs` 第一个函数开始逐行翻译 → 先建立 coarse key 12 种模式 → 再实现 contradiction 分裂 → 最后补齐增量更新和 layer feedback
2. **Causal 重写路径：** 先建立 `record_index → hypothesis → belief` 反向索引 → 实现三种 edge 提取器（explicit / explicit causal connection / temporal） → 聚合为 belief-level patterns → 评分 + evidence gates
3. **Concept 增强路径：** 先实现 CanonicalFeature 的词法层（stemming + equivalence dict + 停用词） → 再实现 cluster guards（tag barrier / family guard / bridge） → 最后对齐 surface 函数
4. **Policy 增强路径：** 先扩展 discover 签名为三引擎输入 → 实现 polarity 分类器 → 实现 action mapping → 实现 4 维 policy_strength 评分 → 最后加 suppression + recommendation 文本
5. 每个引擎修复完成后，更新 AUDIT-DIFF.md 中对应项的修复状态
</specifics>
