# Phase 06.2: MaintenanceService + EpistemicRuntime 重构 - Context

**Gathered:** 2026-05-28
**Status:** Ready for planning

<domain>
## Phase Boundary

将维护编排（写侧）与认知检查（读侧）分离，对齐 Rust 的 `MaintenanceService` + `EpistemicRuntime` 双层架构：

1. **MaintenanceService (新增):** 从 EpistemicRuntime 中移出 `maintain()`，实现完整的维护周期编排（4 阶段 + telemetry + trend + reflection），逐行对齐 Rust `maintenance_service.rs`
2. **EpistemicRuntime (重构):** 变为纯只读认知检查 surface，实现 Rust `epistemic_runtime.rs` 的全部 20+ inspection 方法
3. **Surface 辅助函数 (新增):** `@aura/concept` 和 `@aura/policy` 各新增 surface 函数，对齐 Rust `concept.rs` 和 `policy.rs` 的 surface 模块

**不在本 Phase:**
- 不实现 Rust 的 decay/reflect/consolidation/archival 算法逻辑（这些阶段的方法签名和类型已投影，但算法实现需要依赖 TagTaxonomy、NGramIndex 等尚未在 TS 中完成的模块）
- 不改变引擎内部发现逻辑（BeliefEngine/ConceptEngine/CausalEngine/PolicyEngine 的 discover 方法保持不变）
</domain>

<decisions>
## Implementation Decisions

### 架构分离
- **D-01:** `maintain()` 从 EpistemicRuntime.Interface 中直接移除（破坏性变更），迁移至新的 MaintenanceService
- **D-02:** EpistemicRuntime 变为纯只读 inspection surface，对齐 Rust 的 17 个公开方法

### 语义投影策略
- **D-03:** 逐行翻译 Rust → TypeScript（方法签名 + 计算逻辑 + 常量值完全一致）
- **D-04:** 先产出完整 PROJECTION.md（Rust struct → TS type 对照表），再计划实现

### MaintenanceService 设计
- **D-05:** 放在 `@aura/core`（对齐 Rust 中 `Aura::run_maintenance()` 调用 `MaintenanceService` 的内部编排模式）
- **D-06:** 作为 Effect 函数（`Effect.gen`），非 Context.Tag。通过 Effect.Effect<R, E, EngineDeps> 声明依赖，调用方通过 `Effect.provide(Layer)` 注入
- **D-07:** 语义投影覆盖全部 14 个方法（runInitialPhases, buildSdrLookup, computeLayerStability, runDiscoveryPhases, runPostDiscoveryPhases, finalizeTelemetry, buildTrendSnapshot, pushTrendSnapshot, summarizeTrends, buildReflectionSummary, pushReflectionSummary, summarizeReflections + 常量），即使算法逻辑暂不全部实现

### EpistemicRuntime 设计
- **D-08:** 保持 `@aura/epistemic-runtime` 包名不变
- **D-09:** 全量 1:1 对齐 Rust 的 20+ inspection 方法（Belief 层 6 + Concept 层 4 + Causal 层 1 + Policy 层 6）
- **D-10:** Telemetry 计数器使用 Effect `Ref` 建模（对齐 Rust AtomicU64），在构造函数注入
- **D-11:** surface_concepts/surface_policy_hints 实现在对应引擎包（`@aura/concept`、`@aura/policy`），EpistemicRuntime 委托调用

### 维护配置
- **D-12:** MaintenanceConfig 作为可选参数传入（有默认值），不通过 Effect Context。全部 8 个字段 + 默认 ArchivalRule 全部映射

### 并行策略
- **D-13:** concept discovery 和 causal discovery 通过 `Effect.all(..., { concurrency: 2 })` 并行执行（对齐 Rust rayon::join）

### 实现优先级
- **D-14:** Phase 1: 类型投影（@aura/contract 新增 type/interface）→ Phase 2: EpistemicRuntime inspection 实现 → Phase 3: MaintenanceService 编排实现

### 测试策略
- **D-15:** EpistemicRuntime inspection 方法 → Rust fixture 驱动跨语言一致性测试
- **D-16:** MaintenanceService 编排 → Rust fixture（maintenance cycle 报告）+ TS mock 单元测试
- **D-17:** Surface 函数 → TS 单元测试 + Rust fixture 验证

### Claude's Discretion
- 具体测试用例的边界条件设计
- 每个方法的实现细节（在签名和算法已锁定的前提下）
- Phase 内 plan 的拆分粒度
</decisions>
