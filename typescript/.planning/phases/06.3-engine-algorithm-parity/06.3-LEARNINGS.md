# Phase 06.3 Learnings: Engine Algorithm Parity

## Decisions

### D-06: Contradiction-based hypothesis split replaces Union-Find clustering
BeliefEngine hypothesis generation switched from Union-Find coarse grouping to Rust's `splitByContradiction` approach — supporting/opposing clusters with max 2 hypotheses per belief. This is a fundamental algorithm change, not a parameter tweak.
**Rationale:** Rust belief.rs uses contradiction detection as the primary hypothesis formation mechanism; Union-Find was a placeholder.
**Source:** 06.3-02-SUMMARY.md

### D-07: Deterministic hypothesis IDs via xxhash bridge
`deterministicHypothesisId` uses xxh64 (from `xxhash-wasm`) instead of Rust's xxh3_64. Marked as NON-PARITY — the hash function differs but the determinism property is preserved.
**Rationale:** xxh3_64 is not available in any JS wasm binding; xxh64 provides the same deterministic guarantee with compatible collision resistance.
**Source:** 06.3-02-SUMMARY.md, NON-PARITY comment in BeliefEngine.ts

### D-08: CanonicalFeature as default similarity mode
ConceptEngine now defaults to CanonicalFeature (stemming + equivalence dictionary + Jaccard) instead of SdrTanimoto. Matches Rust `ConceptEngine::new()`.
**Rationale:** Rust uses canonical tokens as the primary concept formation path; SDR is secondary.
**Source:** 06.3-05-SUMMARY.md

### D-09: 3-engine discover signature with Interface types
PolicyEngine.discover accepts `(CausalEngine.Interface, ConceptEngine.Interface, BeliefEngine.Interface, records)` — using contract Interface types instead of concrete *Impl. MaintenanceService wiring updated accordingly.
**Rationale:** Effect DI requires Tag-based service identification; Interface pattern enables mock substitution in tests.
**Source:** 06.3-09-SUMMARY.md

### D-10: README-driven plan correction for scoring formulas
The PLAN.md contained formula guesses (support_score, temporal_consistency, outcome_stability) that diverged from Rust. All 4 were corrected during implementation by grep-verifying against the actual Rust source rather than trusting the plan.
**Rationale:** Plans are hypotheses; implementation must verify against ground truth (Rust source).
**Source:** 06.3-08-SUMMARY.md

Plan formulas for `score_pattern` had 4 errors (support_score formula, temporal_consistency pair source, outcome_stability cv vs variance, missing confidence component). Discovery happened during implementation when grep-verifying against Rust source. The plan was directionally correct but algorithmically imprecise.
**Context:** 06.3-08 executor found and corrected all 4 discrepancies against Rust causal.rs during scoring implementation.
**Source:** 06.3-08-SUMMARY.md

The phase 06.3 research focused on engine code but missed checking MaintenanceService's 5 placeholder `unknown` types. **Context:** MaintenanceService.ts:40-56 had 5 `type X = unknown` placeholders; `SDRInterpreter` exists at `@aura/recall`, `CognitiveStoreFile` at `@aura/storage`.
**Source:** Conversation discovery (not in any SUMMARY.md)

**Context:** Wave 9 cleanup required stash-then-merge; post-phase cleanup had 3 orphaned worktrees.
**Source:** Orchestrator execution logs

## Patterns

### Contract Interface + implements pattern for engine testability
All 4 engines now use `namespace.Interface { readonly discover: ...; readonly stats: ... }` in the contract package, with implementation classes explicitly `implements X.Interface`. This enables `Layer.succeed(Engine.Interface, mockEngine)` in tests without `as any` casts.
**When to use:** Any Effect service that needs mock substitution in tests. Applied to BeliefEngine, CausalEngine, ConceptEngine, PolicyEngine.
**Source:** CR-01 through CR-07 commits from gsd-code-review --fix

### TDD RED-GREEN commit pairs per task
Each implementation task was committed as a RED commit (`test(...): add failing tests`) followed by a GREEN commit (`feat(...): implement`). This produced 22 TDD commit pairs across 10 plans, enabling per-task revert and bisect.
**When to use:** All behavior-adding tasks. Pure doc/config tasks are exempt.
**Source:** 06.3-02 through 06.3-10 SUMMARY.md files

### Enum replacement for string literals in cross-package contracts
String literal unions (`"Positive" | "Negative" | "Neutral"`) in polarity, edge kind, and action kind were replaced with TypeScript const enums (`Polarity.Positive`, `CausalEdgeKind.Explicit`, `PolicyActionKind.Avoid`). This catches typos at compile time and matches the Rust enum pattern one-to-one.
**When to use:** Any cross-package type that maps to a Rust enum in the source of truth.
**Source:** WR-04, WR-06 review fix commits

### Grep-verified constant parity as verification method
Instead of runtime E2E tests for every constant, per-engine type-level parity was verified via `grep` against Rust source for all 19 constants/thresholds/formulas. This is faster than running binaries and catches discrepancies at the code level before tests run.
**When to use:** When Rust source is the ground truth and TS must match constant-for-constant.
**Source:** 06.3-11-SUMMARY.md

## Surprises

### Rust fixture binaries exist but were missed by the verifier
The 06.3-11 executor reported "fixture binaries not found" because it searched `target/release/` (Linux release convention) without `.exe` extension. All 6 binaries existed at `target/debug/*.exe` (Windows debug convention). The `Recall.parity.test.ts` was already passing via `cargo run` the entire time.
**Impact:** False-negative in verification report; must-have #6 was marked ATTEMPTED instead of PASS until a second agent corrected it.
**Source:** 06.3-11-SUMMARY.md, fixture E2E subagent results

### Cross-engine type errors: 36 breakages from contract field additions
Adding 6+ fields to CausalPattern, PolicyHint.cause_record_ids, and Record.connection_types caused 36 type errors across 9 files — all in test mocks and factory functions. The engines themselves compiled fine; the breakage was entirely in test infrastructure.
**Impact:** Plan 06.3-11 Task 1 was almost entirely test mock updates rather than implementation fixes.
**Source:** 06.3-11-SUMMARY.md

### The phase scope was correct but the MaintenanceService gap was invisible
Phase 06.3's 11 plans correctly covered all 4 engines' algorithm parity. But MaintenanceService's 14 D-07 deferred stubs and 5 placeholder types — sitting between the engines and the full pipeline — were never flagged by any plan or verification step. The gap only surfaced during user review.
**Impact:** 20+ TODOs discovered post-execution; 3 captured as todos for the next phase.
**Source:** Conversation discovery

### Review fix iteration count exceeded expectations
What was planned as a single code-review pass turned into 3 rounds of review fixes: the initial `gsd-code-review --fix` (17 commits on `gsd-reviewfix/06.3-709`), a second round of warning fixes (WR-01~05), and a third round of info fixes (IN-01~06). Total: 25+ fix commits after the 11 plans were "complete".
**Impact:** Phase duration extended by approximately 2x beyond the planned execution time. The code quality improvement was real but the process was not anticipated.
**Source:** Git log, REVIEW-FIX.md
