# Phase 06.1 Learnings: 补齐四大引擎功能 + 类型修复

## Decisions

### D1: Contract types drive implementation types
When the contract says `Effect<A, never, EpistemicTrace>` but the implementation needs `BeliefEngine | ConceptEngine | CausalEngine | PolicyEngine` in R, the fix is to widen the contract — not suppress types with `as any`. This revealed that EpistemicRuntime is genuinely an orchestrator that needs all four engines, which the old contract hid.

**Rationale:** TypeScript strict mode + Effect-TS R tracking catches real architectural issues. Widening R types in the contract documents actual dependencies rather than hiding them.
**Source:** 06.1-01-SUMMARY.md, 06.1-CONTEXT.md

## Lessons

### L1: `replace_all: true` on Edit tool can miss patterns
`result[0][1]` appears twice in a test file with two different expected values (`"r1"` and `"r3"`). Using `replace_all: true` only matches exact string equality — it changed `"r1"` but missed `"r3"` because the surrounding context was different. Always grep after replace_all to verify coverage.

**Context:** BoundedReranker.test.ts had two `result[0][1]` assertions with different trailing strings. replace_all caught one but not the other.
**Source:** 06.1-01-SUMMARY.md

### L2: Smart quotes corrupt TypeScript source on Windows
Edit operations that touch files with mixed encoding can introduce Unicode smart quotes (`“` `”`) instead of ASCII `"`, producing `TS1127: Invalid character` errors. The safe recovery is `git checkout HEAD -- <file>` and rewrite with Write tool. Prefer Write over Edit for large multi-section replacements.

**Context:** BeliefEngine.ts had Chinese comments — an Edit that matched a large block introduced smart quotes throughout the file, breaking all string literals and enum values.
**Source:** 06.1-01-SUMMARY.md (execution notes)

## Patterns

### P1: Type error triage by package
When fixing type errors across packages, group by package and fix in dependency order: contract types first → engine implementations → tests last. Running `bun run typecheck | grep "^packages/" | grep -v code-extraction` after each package fix provides fast feedback.

**When to use:** Any multi-package type error fixing session.
**Source:** 06.1-01-SUMMARY.md

## Surprises

### S1: `Effect.dieMessage` doesn't exist in effect@4.0.0-beta.68
The expected API for creating defect-based Effects is `Effect.die(new Error(msg))`, not `Effect.dieMessage(msg)`. The dieMessage variant was added in a later Effect version. This affected CausalEngine and PolicyEngine stub implementations during the contract typing phase.

**Impact:** Switched all stubs from `Effect.dieMessage("not implemented")` to `Effect.die(new Error("not implemented"))`. This is a beta-version API difference worth documenting for the project.
**Source:** 06.1-01-SUMMARY.md
