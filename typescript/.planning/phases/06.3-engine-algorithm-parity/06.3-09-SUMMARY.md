## Task Summary

### Task 1: 3-engine discover signature (Interface types) + seed selection (6 conditions) + provenance

**RED commit:** `3e37561` — `test(06.3-09): add failing tests for 3-engine discover + seed selection (RED)`
**GREEN commit:** `a71cb46` — `feat(06.3-09): implement 3-engine discover with seed selection (6 gates) + provenance (GREEN)`

**What was done:**

1. **ConceptEngine.Interface namespace** — Added `ConceptEngine.Interface` following the same pattern as `CausalEngine.Interface` and `BeliefEngine.Interface`. Old `ConceptEngineImpl` type preserved as deprecated alias for backward compatibility. Updated `Policy.ts` contract to use `ConceptEngine.Interface` in the PolicyEngine discover signature.

2. **3-engine discover signature** — `discover(causal_engine: CausalEngine.Interface, concept_engine: ConceptEngine.Interface, belief_engine: BeliefEngine.Interface, records)` — all three engines use Interface types, NOT *Impl.

3. **selectSeeds function** — Pure function applying 6 gates to filter causal patterns:
   - **Gate 1 — Strength:** Stable passes always; Candidate needs causal_strength >= 0.65; ExplicitTrusted bypass for explicit_support >= 1 with non-Rejected/Invalidated state
   - **Gate 2 — Support:** support_count >= 2 OR ExplicitTrusted with explicit_support >= 1
   - **Gate 3 — Evidence:** meetsEvidenceGate from @aura/causal — tries StrictRepeatedWindows first, falls back to ExplicitTrusted, then TemporalClusterRecovery (with strict constraints: explicit=0, counterevidence=0, positive_signals>0, negative_signals=0)
   - **Gate 4 — Counterevidence:** counterevidence_count <= support_count / 2 (local pure helper)
   - **Gate 5 — Counterfactual:** meetsCounterfactualGate from @aura/causal (ratio check + dominance + consistency)
   - **Gate 6 — Belief:** At least one cause belief must be Resolved/Singleton, OR ExplicitTrusted bypass

4. **Provenance fields** — `PolicySeed` type includes `cause_record_ids: string[]` and `effect_record_ids: string[]` copied from CausalPattern.

5. **TDD:** 16 RED tests (stub returns 0) → 16 GREEN (all pass). Tests cover: strength gate (Stable/Candidate>=0.65/Candidate<0.65/ExplicitTrusted bypass/Rejected), support gate (>=2/1), counterevidence gate, belief gate (Resolved/Singleton/Unresolved/missing), evidence gate, provenance, multi-pattern filtering.

### Task 2: Polarity classification + action mapping + MaintenanceService update

**RED commit:** `605348f` — `test(06.3-09): add failing tests for polarity classification + action mapping (RED)`
**GREEN commit:** `9c792e2` — `feat(06.3-09): implement polarity classification + action mapping + MaintenanceService update (GREEN)`

**What was done:**

1. **polaritySignalCounts** — Pure function counting positive/negative signals from effect records across 5 dimensions:
   - `semantic_type === "contradiction"` → negativeSignals += 2
   - Tags checked against NEGATIVE_KEYWORDS (15 items, case-insensitive contains) → +1 per match
   - Tags checked against POSITIVE_KEYWORDS (14 items, case-insensitive contains) → +1 per match
   - Content checked against NEGATIVE_KEYWORDS → +1 per match
   - Content checked against POSITIVE_KEYWORDS → +1 per match
   - Summed across all effect records

2. **classifyPolarity** — Returns "Negative" (negSig > posSig && negSig >= 2), "Positive" (posSig > negSig && posSig >= 2), or "Neutral" (otherwise). Matches Rust exactly.

3. **mapActionKind** — Deterministic lookup table:
   | Polarity + Strength | Action Kind |
   |---------------------|-------------|
   | Negative + >= 0.75  | Avoid |
   | Negative            | VerifyFirst |
   | Positive + >= 0.75  | Prefer |
   | Positive            | Recommend |
   | Neutral             | Warn |

4. **MaintenanceService cleanup** — Removed `const causalStateForPolicy = yield* causalEngine.stats()` (line 665) — the policy engine now fetches causal state internally via discover. The discover call already passes 3 engines (causalEngine, conceptEngine, beliefEngine).

5. **TDD:** 9 RED tests (stub returns wrong values) → 9 GREEN (all pass). Tests cover: Negative/Positive/Neutral classification, signal counting (single/multi-record), action mapping for all 5 kinds.

## Known Stubs

1. **P2 hint construction** — The discover method counts seeds but does not yet build PolicyHints from them. This is deferred to P2 (4-dim policy_strength scoring, suppression, recommendation templates). The P1 implementation validates all 6 gates and returns correct seeds_found count.

2. **believe_for_record fallback** — The belief gate uses `beliefState.beliefs[causeBid]` for checking cause belief state. In production, there may be records without belief mappings; the current implementation handles this gracefully (null cause_belief_id → belief gate fails unless ExplicitTrusted bypass).
