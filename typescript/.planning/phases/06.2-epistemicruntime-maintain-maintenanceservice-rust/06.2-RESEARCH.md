## Summary

This phase separates maintenance orchestration (write-side) from epistemic inspection (read-side), aligning with Rust's `MaintenanceService` + `EpistemicRuntime` dual-layer architecture. The work is split into three implementation phases:

1. **Type projection:** ~30 new TypeScript types across `@aura/contract` (new files `Maintenance.ts`, `EpistemicInspection.ts`) plus updates to existing types
2. **EpistemicRuntime refactor:** Remove `maintain()`, add 20+ read-only inspection methods, migrate to `namespace.Interface` pattern
3. **MaintenanceService (new):** 14 Effect functions for full maintenance cycle orchestration in `@aura/core`

**Primary recommendation:** Follow the existing codebase patterns (Effect.gen, contract-interface-pattern, serviceOption) and line-by-line map the Rust source. The PROJECTION.md is the authoritative method-by-method specification.

**Data flow (maintenance cycle):**
```
External trigger -> Aura.runMaintenance()
  -> MaintenanceService.runInitialPhases() [level_fix -> decay -> reflect -> epistemic_state -> insights]
  -> MaintenanceService.buildSdrLookup() [SDR vector cache management]
  -> MaintenanceService.computeLayerStability() [key_index diff across layers]
  -> MaintenanceService.runDiscoveryPhases() [belief -> concept||causal -> policy -> feedback]
  -> MaintenanceService.runPostDiscoveryPhases() [consolidation -> cross_connections -> archival]
  -> MaintenanceService.finalizeTelemetry() [dominant_phase + concept surface telemetry]
  -> MaintenanceService.buildTrendSnapshot() + pushTrendSnapshot()
  -> MaintenanceService.buildReflectionSummary() + pushReflectionSummary()
  -> return MaintenanceReport
```

**Data flow (epistemic inspection):**
```
External query -> EpistemicRuntime.getBeliefs/getConcepts/...
  -> reads engine state via Effect.service(Engine)
  -> filter/sort/compute aggregates
  -> return typed result
```

### Pattern 2: Effect Function Composition (MaintenanceService)
**What:** Service functions as exported Effect.gen generators, NOT as Context.Tag interface
**When to use:** Internal orchestration that does not need service polymorphism; callers invoke directly and provide dependencies
**Example:**
```typescript
// @aura/core/src/MaintenanceService.ts
export function runDiscoveryPhases(
  beliefEngine: BeliefEngine.Interface,
  beliefStore: BeliefStoreImpl,
  conceptEngine: ConceptEngineImpl,
  // ...
): Effect.Effect<DiscoveryPhaseResult, never,
  EpistemicTrace | BeliefEngine | ConceptEngine | CausalEngine | PolicyEngine
    | BeliefStore | ConceptStore | CausalStore | PolicyStore> {
  return Effect.gen(function* () {
    // Phase 3.5: Belief
    const belief = yield* beliefEngine.update_with_sdr(beliefSnapshot, sdrLookup)
    yield* beliefStore.save(stuff)

    // Phase 3.7 & 3.8: Concept + Causal (parallel)
    const [concept, causal] = yield* Effect.all({
      concept: runConceptPhase(conceptEngine, conceptStore, beliefSnapshot, sdrLookup),
      causal: runCausalPhase(causalEngine, causalStore, beliefSnapshot, sdrLookup)
    }, { concurrency: 2 })

    // Phase 3.9: Policy
    const policy = yield* runPolicyPhase(policyEngine, policyStore, /* ... */)

    // Phase Feedback
    const feedback = yield* runFeedbackPhase(beliefEngine, causalEngine, policyEngine, beliefStore)

    return { belief, concept, causal, policy, feedback }
  })
}
```

### Pattern 3: Ref-based Telemetry Counters
**What:** Use Effect `Ref` for thread-safe mutable counters (aligns Rust AtomicU64)
**When to use:** Telemetry/observability counters that need atomic read-and-reset each cycle
**Example:**
```typescript
import { Ref, Effect } from "effect"

// Constructor
constructor(
  private readonly conceptSurfaceMode: ConceptSurfaceMode,
  private readonly globalCalls: Ref.Ref<number>,
  private readonly namespaceCalls: Ref.Ref<number>,
  // ...3 more Refs
) {}

// Usage in tracking method
private trackGlobalCall(returned: number): Effect.Effect<void> {
  return Effect.gen(function* () {
    yield* Ref.update(this.globalCalls, (n) => n + 1)
    yield* Ref.update(this.conceptsReturned, (n) => n + returned)
  })
}

// Reset at end of maintenance cycle (swap reads current value and sets to 0)
const calls = yield* Ref.getAndSet(this.globalCalls, 0)
```
[ASSUMED: based on D-10 and Effect Ref API in Effect 4.0.0-beta.68]

### Example: Surface Function Pattern
**What:** Filter -> sort -> map pipeline for surfacing concepts/policy hints
**Example:**
```typescript
// @aura/concept/src/Surface.ts
export function surfaceConcepts(
  engine: ConceptEngineImpl,
  limit?: number
): Effect.Effect<ReadonlyArray<SurfacedConcept>> {
  return Effect.gen(function* () {
    const state = yield* engine.stats()
    const max = limit ?? 20

    // Phase A: filter eligible (Stable or Candidate)
    // Phase B: sort by abstractionScore desc
    // Phase C: limit and map to SurfacedConcept
    const candidates = Object.values(state.concepts)
      .filter(c => c.state === ConceptState.Stable || c.state === ConceptState.Candidate)
      .sort((a, b) => b.abstraction_score - a.abstraction_score)
      .slice(0, max)

    return candidates.map(c => ({
      id: c.id,
      key: c.key,
      state: c.state,
      namespace: c.namespace,
      abstractionScore: c.abstraction_score,
      beliefCount: c.belief_ids.length,
      recordCount: c.record_ids.length,
      coreTerms: c.core_terms,
      recordIds: c.record_ids,
    }))
  })
}
```

### Anti-Patterns to Avoid
- **Re-creating Effect context Tag for MaintenanceService:** D-06 explicitly says MaintenanceService functions are NOT Tags, they are exported functions that accept dependencies as parameters. Do not create a `MaintenanceService` Tag.
- **Inline algorithm logic in EpistemicRuntime that belongs to engine packages:** D-11 delegates surface logic to engine packages. EpistemicRuntime only calls the surface functions, does not re-implement them.
- **Hardcoded MaintenanceConfig in multiple places:** Use the `defaultMaintenanceConfig` constant in one place and pass as parameter (D-12).
- **Using Context to access MaintenanceConfig:** D-12 says config is an optional param with defaults, not via Effect Context.

## Common Pitfalls

### Pitfall 1: Package dependency cycle via @aura/contract
**What goes wrong:** `@aura/core` depends on `@aura/epistemic-runtime` for DefaultLayer, and `@aura/epistemic-runtime` depends on `@aura/core` for types. Circular dependency breaks.
**Why it happens:** MaintenanceService is in `@aura/core` and needs engine type references. EpistemicRuntime in `@aura/epistemic-runtime` is already wired in `@aura/core/DefaultLayer.ts`.
**How to avoid:** MaintenanceService should only depend on `@aura/contract` types and Effect — NOT on `@aura/epistemic-runtime`. Use `@aura/contract` for all shared types. EpistemicRuntimeImpl's Interface stays in `@aura/contract`.
**Current state:** `@aura/core/package.json` has dependencies on `@aura/contract`, `@aura/storage`, `@aura/belief`, `@aura/concept`, `@aura/causal`, `@aura/policy`, `@aura/epistemic-runtime`, `@aura/recall`, `@aura/utils`. This is correct — `@aura/core` is the integration layer.

### Pitfall 2: Modeling Rust mutability patterns in Effect
**What goes wrong:** Rust's `&mut` mutation (e.g., `records: &mut HashMap<String, Record>`) gets naively translated to mutable let variables in TS, breaking referential transparency.
**Why it happens:** Rust's ownership model allows scoped mutation. Effect.gen functions should use `yield* Ref.update()` or pass state explicitly.
**How to avoid:** Where Rust takes `&mut records`, the TS equivalent should either:
- For local state within a single Effect.gen: use mutable `let` variables (acceptable, scoped within the generator)
- For shared mutable state across function calls: use `Ref`
- For sequential data transformation: use immutable data flow (map/filter/reduce)

### Pitfall 3: Complex aggregate methods require heavy computation in Effect
**What goes wrong:** `getBeliefInstabilitySummary()`, `getContradictionClusters()`, `getPolicyLifecycleSummary()` iterate all beliefs/hints and perform sorting/grouping. In Rust this is fast with direct RwLock access. In Effect it must yield to each engine service call.
**How to avoid:** Load state once at the start of each method via `yield* engine.stats()` (returns the full engine state snapshot), then perform all computation locally on the snapshot. This matches the Rust pattern of acquiring the read lock once and iterating.

### Pitfall 4: Inconsistent type naming between Rust and TS conventions
**What goes wrong:** Rust uses `snake_case` fields; TS convention in the codebase uses `camelCase` for runtime types but snake_case for persisted/contract types.
**How to avoid:** Follow existing convention in the codebase:
- Contract types (matching Rust structs for serialization): use `snake_case` field names to match Rust disk format
- New TS-only interfaces/types: use `camelCase`
- This is already established in `BeliefTypes.ts`, `ConceptTypes.ts`, etc.

### Pitfall 5: Forgetting to update DefaultLayer.ts
**What goes wrong:** After refactoring, `DefaultLayer.ts` still has the old EpistemicRuntimeLive with maintain() capabilities.
**How to avoid:** EpistemicRuntimeLive stays (D-08 says keep the package), but the Live Layer registration in `DefaultLayer.ts` must be updated to reflect the new EpistemicRuntimeImpl constructor signature (now takes ConceptSurfaceMode + 5 Refs). The Live Layer should provide defaults for these (Inspect mode, zero Refs).

   - Recommendation: Phase 06.2 should add the missing fields to the TS PolicyHint type in `PolicyTypes.ts` (specifically: `actionKind`, `policyStrength`, `riskScore`, `namespace`, `domain`, `key`). The PROJECTION.md already defines `PolicyActionKind` enum. This is a necessary dependency for EpistemicRuntime inspection methods.

2. **`Belief` type missing `key` field?**
   - What we know: Rust Belief has `key: String` field, used by `getContradictionClusters` for belief key grouping. TS `Belief` type (`@aura/contract/src/belief/BeliefTypes.ts`) does NOT have a `key` field.
   - What's unclear: Is `key` implicitly accessible? Does the Rust `key_index` serve the same purpose?
   - Recommendation: Verify that TS `Belief` type needs a `key` field added for inspection methods. If `key_index` is separate (it is in `BeliefEngineState`), either add key to Belief type or pass key_index alongside.

3. **`Rust epistemic_runtime.rs` records field — how does TS EpistemicRuntime get records?**
   - What we know: Rust EpistemicRuntime holds `records: RwLockReadGuard<HashMap<String, Record>>` for record-level lookups in `getContradictionClusters` and `getSurfacedConceptsForRecord`.
   - What's unclear: The TS EpistemicRuntime does not currently receive records. Should records be passed as a parameter to relevant methods, or stored as a field like Rust?
   - Recommendation: Pass records as a parameter to methods that need them (getContradictionClusters, getSurfacedConceptsForRecord), similar to how Rust holds a read lock. This avoids adding lifetime management issues.

4. **Effect `Ref` as constructor parameter vs `Layer.effect` wiring**
   - What we know: Rust creates `AtomicU64` counters inline. TS needs something similar.
   - What's unclear: Should the EpistemicRuntime live Layer create these Refs and inject them via constructor parameters?
   - Recommendation: Use `Layer.effect(EpistemicRuntime, ...)` to create the Refs (via `Ref.make(0)`) and construct EpistemicRuntimeImpl with them. This is the cleanest pattern that matches Effect's resource lifecycle.
