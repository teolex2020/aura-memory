# Phase 06.2 Learnings: EpistemicRuntime/MaintenanceService Separation (Rust Alignment)

## Decisions

### D-01: maintain() removed from EpistemicRuntime.Interface entirely
The `maintain()` method is removed from the EpistemicRuntime contract interface and migrated to a new MaintenanceService. EpistemicRuntime becomes a pure read-only inspection surface with 18 methods across 4 layers (6 Belief + 4 Concept + 1 Causal + 7 Policy).

**Rationale:** Aligns with Rust's dual-layer architecture where MaintenanceService handles write-side orchestration and EpistemicRuntime handles read-side inspection. This is a breaking change -- callers must use the new MaintenanceService for maintenance cycles.

**Source:** 06.2-CONTEXT.md D-01, 06.2-01-PLAN.md Task 3, 06.2-VERIFICATION.md Truth 3

---

### D-03: Line-by-line Rust to TypeScript semantic projection
Method signatures, computation logic, and constant values are projected line-by-line from Rust source files (maintenance_service.rs, epistemic_runtime.rs, background_brain.rs, concept.rs surface, policy.rs surface) to TypeScript. This includes all field names, enum values, and default constant values.

**Rationale:** Ensures 1:1 behavioral alignment between the TypeScript port and the Rust reference implementation. Field names adapt to camelCase (TypeScript convention) from Rust's snake_case for runtime types, but the structure and semantics are identical.

**Source:** 06.2-CONTEXT.md D-03, 06.2-01-PLAN.md (objective, Task 1 action)

---

### D-05: MaintenanceService placed in @aura/core, not @aura/epistemic-runtime
The MaintenanceService is placed in the @aura/core package, matching Rust's pattern where Aura::run_maintenance() calls MaintenanceService as an internal orchestration helper. The 14 Effect functions are exported individually from @aura/core/src/MaintenanceService.ts.

**Rationale:** The MaintenanceService orchestrates all four engine services (Belief, Concept, Causal, Policy) plus stores. Placing it in @aura/core gives it access to DefaultLayer and the Aura facade, where runMaintenance is exposed. @aura/epistemic-runtime remains focused only on inspection methods.

**Source:** 06.2-CONTEXT.md D-05, 06.2-06-PLAN.md (objective)

---

### D-06: MaintenanceService as Effect functions, not Context.Tag
The 14 MaintenanceService functions are exported as individual Effect functions (Effect.gen) that accept engine instances as parameters, not as Context.Tag services. Callers provide engines and stores explicitly via function arguments.

**Rationale:** Unlike EpistemicRuntime which is a Context.Tag service obtained via Effect.service(), MaintenanceService functions are pure orchestration helpers. They accept specific engine instances, making them independently testable without Layer.provide. The function signatures declare their Effect requirements via the generic R type parameter.

**Source:** 06.2-CONTEXT.md D-06, 06.2-06-PLAN.md (Tasks 1-2)

---

### D-07: All 14 method signatures projected even with deferred algorithm logic
All 14 MaintenanceService function signatures and 5 constants are defined, even when the algorithm implementations are stubs (marked with TODO D-07 comments). This includes runInitialPhases, buildSdrLookup, computeLayerStability, runDiscoveryPhases, runPostDiscoveryPhases, finalizeTelemetry, buildTrendSnapshot, pushTrendSnapshot, summarizeTrends, buildReflectionSummary, pushReflectionSummary, and summarizeReflections.

**Rationale:** Establishing the full function surface area upfront enables the orchestration pipeline in Aura.runMaintenance to compile and test with mock engines. Algorithm-heavy sub-phases (decay, reflect, consolidation, archival, reflection findings) are deferred until their dependencies (TagTaxonomy, NGramIndex, CognitiveStore) exist in TypeScript.

**Source:** 06.2-CONTEXT.md D-07, 06.2-06-PLAN.md (Tasks 1-2, stub comments)

---

### D-10: Telemetry counters use Effect Ref (matching Rust AtomicU64)
Concept surface telemetry counters (globalCalls, namespaceCalls, recordCalls, conceptsReturned, recordAnnotationsReturned) are modeled as Effect `Ref.Ref<number>` values injected via the EpistemicRuntimeImpl constructor. They are created in EpistemicRuntimeLive (Layer.effect) and updated atomically via Ref.update.

**Rationale:** Rust uses AtomicU64 for telemetry counters. Effect Ref provides equivalent atomic read-modify-write semantics with thread-safe updates in the Effect runtime. The Ref is created at Layer initialization time and shared across all inspection method calls.

**Source:** 06.2-CONTEXT.md D-10, 06.2-04-PLAN.md (Task 1 action)

---

### D-11: Surface functions implemented in engine packages, not EpistemicRuntime
surfaceConcepts/surfaceConceptsFiltered are implemented in @aura/concept. surfacePolicyHints/surfacePolicyHintsFiltered are implemented in @aura/policy. EpistemicRuntime delegates to these functions, importing them as workspace dependencies.

**Rationale:** Surface functions operate on engine state (ConceptEngineState, PolicyHint arrays) which lives in the respective engine packages. Placing surface logic in the engine package follows the single-responsibility principle and keeps EpistemicRuntime as a thin delegation layer. @aura/epistemic-runtime adds explicit workspace dependencies on @aura/concept and @aura/policy.

**Source:** 06.2-CONTEXT.md D-11, 06.2-02-PLAN.md (feature), 06.2-03-PLAN.md (feature)

---

### D-12: MaintenanceConfig as optional parameter with defaults
MaintenanceConfig is passed as an optional function parameter (not via Effect Context) with `defaultMaintenanceConfig` fallback. The default config includes exact 10-item archivalRules array and 2-item completedArchivalRules array, matching Rust Default impl.

**Rationale:** MaintenanceConfig is a static value that callers set once per maintenance cycle. Using a function parameter is simpler than a Context.Tag for this use case. The default config provides reasonable defaults for all 8 fields, with the defaultMaintenanceConfig const matching the Rust Default impl exactly.

**Source:** 06.2-CONTEXT.md D-12, 06.2-01-PLAN.md Task 1 (defaultMaintenanceConfig)

---

### D-13: Concept and Causal discovery in parallel via Effect.all concurrency:2
The runDiscoveryPhases function runs conceptEngine.discover() and causalEngine.discover() in parallel using `Effect.all({ concept: ..., causal: ... }, { concurrency: 2 })`. Belief discovery runs first (sequentially), then concept+causal in parallel, then policy sequentially after both complete.

**Rationale:** Rust uses rayon::join for parallel concept and causal discovery. Effect.all with concurrency:2 provides equivalent behavior in the Effect-TS runtime. The belief phase must complete first because concept and causal both depend on belief state.

**Source:** 06.2-CONTEXT.md D-13, 06.2-06-PLAN.md (Task 2, runDiscoveryPhases)

---

### D-14: Type projection -> EpistemicRuntime inspection -> MaintenanceService orchestration
The implementation order follows a strict three-phase sequence: (1) Contract type definitions in @aura/contract, (2) EpistemicRuntime inspection methods and surface functions, (3) MaintenanceService orchestration and Aura wiring.

**Rationale:** Each phase depends on the previous phase's output. Types must exist before implementations can use them. Inspection methods must exist before MaintenanceService can call them (for trend/reflection data). Plans are ordered 01 (types) -> 02/03/04 (inspection + surface) -> 05/06 (aggregate methods + MaintenanceService) to respect this dependency chain.

**Source:** 06.2-CONTEXT.md D-14, 06.2-01-PLAN.md through 06.2-06-PLAN.md (depends_on chains)

---

### GAP-01/02: Verification gaps identified and closed post-implementation
Two verification gaps were identified during the verification pass: (1) Surface.ts defined local PolicyActionKind/PolicyState types instead of importing from @aura/contract, and (2) Aura.runMaintenance was an Effect.die stub instead of being wired to MaintenanceService. Both were fixed in dedicated gap-closure plans.

**Rationale:** The verification process (06.2-VERIFICATION.md) checks key_links between packages. Gap plans were created specifically to close these wiring gaps. GAP-01 fixed type imports in Surface.ts. GAP-02 wired the full Aura.runMaintenance pipeline with 10 MaintenanceService function calls.

**Source:** 06.2-GAP-01-PLAN.md (objective), 06.2-GAP-02-PLAN.md (objective), 06.2-VERIFICATION.md (gaps section)

---

### Effect.Layer.effect for services that need Ref creation
EpistemicRuntimeLive uses `Layer.effect(EpistemicRuntime, Effect.gen(function* () { ... }))` instead of `Layer.succeed()` because the implementation needs to create Ref values at initialization time. The generator creates all 5 Ref values, then constructs the EpistemicRuntimeImpl with them.

**Rationale:** Ref values must be created within the Effect context (via Ref.make()), which requires an Effect, not a plain value. Layer.effect bridges this gap by allowing Effect-based service creation.

**Source:** 06.2-04-PLAN.md Task 1 (EpistemicRuntimeLive implementation), 06.2-VERIFICATION.md Truth 11

## Lessons

### Effect.gen type inference differs across effect versions
Effect v4.0.0-beta.68's Effect.gen does not support the `$` callback parameter pattern. The generator function uses `function* ()` without arguments. Explicit return type annotations conflict with exactOptionalPropertyTypes: true in tsconfig, requiring removal of explicit return types.

**Context:** The plan specified `function* ($)` pattern with `$` parameter, but this version of effect requires `function* ()` without arguments. Also, explicit `Effect.Effect<MaintenanceReport, FileWriteError, ...>` return type annotations conflicted with Effect.gen inference under `exactOptionalPropertyTypes: true`.

**Source:** 06.2-GAP-02-SUMMARY.md (deviations 2-3)

---

### Unused imports cause compilation errors in strict TypeScript
Files that import types that are not actually used (e.g., ConceptSurfaceMode in EpistemicRuntime.ts when it was not referenced in method signatures) cause TypeScript compilation errors. These must be cleaned before typecheck passes.

**Context:** Plan 01 created EpistemicInspection.ts with an unused `import type { ConceptSurfaceMode } from "./Maintenance"` and EpistemicRuntime.ts with unused imports. These caused typecheck failures and were fixed by removing the unused imports.

**Source:** 06.2-01-SUMMARY.md (Auto-fixed Issues 2)

---

### Plan import paths often need correction from actual source locations
Plan documents specify import paths that may be incorrect at implementation time. For example, the plan specified `import type { Belief, BeliefEngine } from "./Belief"` but `Belief` type is actually exported from `./belief/BeliefTypes`. Similar issues existed for ConceptCandidate, CausalPattern, and PolicyHint.

**Context:** During Task 3 of Plan 01, all four type import paths had to be corrected from the plan's defaults to the actual file locations. These are auto-fixed issues that required no scope creep but were necessary for correctness.

**Source:** 06.2-01-SUMMARY.md (Auto-fixed Issues 1)

---

### Project-internal workspace dependencies must be explicitly declared
@aura/epistemic-runtime needed explicit `"@aura/concept": "workspace:*"` and `"@aura/policy": "workspace:*"` in its package.json to import surface functions. These dependencies were not present from Phase 5 because epistemic-runtime previously only depended on @aura/contract.

**Context:** Plan 05 added surface delegation methods that call surfaceConcepts() from @aura/concept and surfacePolicyHints() from @aura/policy. These imports require explicit workspace dependency declarations in the importing package's package.json, plus a workspace-level `bun install` to resolve.

**Source:** 06.2-05-SUMMARY.md (Dependencies section)

---

### Worktree file path management requires explicit derivation
Multiple plans experienced file path issues where edits were applied to the main repo path (`D:/workspace/.../typescript/packages/...`) instead of the worktree path (`D:/workspace/.../.claude/worktrees/agent-*/typescript/packages/...`). The fix requires using `git rev-parse --show-toplevel` to derive the correct worktree path.

**Context:** Both Task 1 of Plan 01 and Task 1 of GAP-01 experienced this issue. The initial commits wrote to the main repo path and had to be corrected by re-applying edits to the worktree version.

**Source:** 06.2-01-SUMMARY.md (Auto-fixed Issues 3), 06.2-GAP-01-SUMMARY.md (Issues Encountered)

---

### satisfies keyword does not narrow Effect.gen return type sufficiently
The `satisfies MaintenanceReport` pattern does not narrow the return type enough for Effect.gen inference when `exactOptionalPropertyTypes: true` is set in tsconfig. The `as MaintenanceReport` cast is required instead.

**Context:** During GAP-02 Task 2, the plan's recommended `satisfies MaintenanceReport` pattern caused TypeScript errors because Effect.gen infers the exact return type shape, and `satisfies` does not narrow it to match the declared Effect type. The fix was to use `as MaintenanceReport`.

**Source:** 06.2-GAP-02-SUMMARY.md (Auto-fixed Issues 3)

---

### Pre-existing type errors can hide behind unmodified files
The GAP-01 summary documents 4 pre-existing type errors in files not modified by the plan (PolicyEngine.ts, PolicyStoreFile.ts, PolicyStoreFile.test.ts, MaintenanceService.test.ts) that existed independently of the gap-closure work. These errors were introduced by earlier plans (06.2-01, 06.2-06) but not caught by typecheck on those plans.

**Context:** The SURFACE.md notes these as "Pre-existing Type Errors (Deferred)" with specific file/error mappings. The PolicyEngine.ts error is missing required fields from the contract PolicyHint (actionKind, policyStrength, etc.) that were added in plan 06.2-01.

## Patterns

### namespace.Interface pattern for service interface definitions
EpistemicRuntime uses `export namespace EpistemicRuntime { export interface Interface { ... } }` matching the BeliefEngine precedent. The Tag class is `export class EpistemicRuntime extends Context.Tag("EpistemicRuntime")<EpistemicRuntime, EpistemicRuntime.Interface>() {}`.

**When to use:** Defining service interfaces that follow the @aura/contract convention. The namespace groups the interface with the Tag class and any helper types, matching the established codebase pattern.

**Source:** 06.2-01-PLAN.md Task 3, 06.2-01-SUMMARY.md (patterns-established)

---

### Layer.effect for services requiring Effect-time initialization
Services that need to create Refs, open connections, or perform async initialization at construction time use `Layer.effect(Tag, Effect.gen(function* () { ... }))` instead of `Layer.succeed(Tag, value)`. The generator creates necessary Effect-based resources before constructing the service instance.

**When to use:** Services whose construction requires Effect operations (Ref.make, Queue, etc.) that cannot be done in a plain constructor. Established in Plan 04 for EpistemicRuntimeLive and reused as a pattern for any service with Ref-based state.

**Source:** 06.2-04-PLAN.md Task 1, 06.2-VERIFICATION.md Truth 11

---

### Surface function pipeline: filter -> sort -> limit -> map
Both concept and policy surface functions follow the same pipeline: (1) Filter eligible items by state (Stable/Candidate), (2) Sort by primary key (abstractionScore or policyStrength desc), (3) Apply limit (default 20), (4) Map to SurfactedConcept or SurfactedPolicyHint. Policy adds an actionKind priority tiebreaker.

**When to use:** Converting internal engine state (ConceptCandidates, PolicyHints) to public-facing view types. The pipeline ensures consistent ordering and bounded results regardless of engine state size.

**Source:** 06.2-02-PLAN.md (feature behavior), 06.2-03-PLAN.md (feature behavior)

---

### Ref-based bounded history for trend and reflection data
MaintenanceService maintains bounded arrays for maintenance trend history (max 32 entries) and reflection history (max 16 entries) using `pushTrendSnapshot` and `pushReflectionSummary` helpers. When the array exceeds the limit, the oldest entry is shifted.

**When to use:** Any rolling window of historical data where only recent entries matter and unbounded accumulation is undesirable. The limits match Rust constants exactly.

**Source:** 06.2-06-PLAN.md Task 1 (pushTrendSnapshot, pushReflectionSummary), 06.2-06-SUMMARY.md (constants)

---

### Engine adapter bridge for surface delegation
EpistemicRuntime's getSurfacedPolicyHints method creates an adapter object that conforms to PolicyEngineImpl's interface (with hints property and values() method) from contract types (PolicyHint[]). This bridges the type gap between the contract PolicyHint shape and the surface module's expected input.

**When to use:** When a module expects engine-typed parameters but the calling module only has contract types. The adapter pattern avoids modifying either module's interface. Same approach applies to concept surface delegation.

**Source:** 06.2-05-SUMMARY.md (Implementation notes - Policy surface adapter)

---

### Shared telemetry counters across service boundaries
ConceptSurfaceCounters type uses Effect Ref values that are shared between EpistemicRuntime (which increments them via track* methods) and MaintenanceService (which reads and resets them via finalizeTelemetry). The Ref is created in @aura/epistemic-runtime's EpistemicRuntimeLive and accessed by MaintenanceService via the Counters parameter.

**When to use:** Cross-service telemetry or metrics that span the boundary between read-side (inspection) and write-side (maintenance) operations. Ref provides atomic read-update semantics without locks.

**Source:** 06.2-01-PLAN.md Task 1 (ConceptSurfaceCounters), 06.2-04-PLAN.md (telemetry methods), 06.2-06-PLAN.md Task 2 (finalizeTelemetry)

## Surprises

### 45 tests for EpistemicRuntime, 181 total across the entire phase
EpistemicRuntime.test.ts ended up with 45 test cases covering all 18 methods, parameter clamping, empty state handling, and filtering. The full phase test suite runs 104 tests across 10 files (concept Surface: 11, policy Surface: 12, core Aura: 2, core MaintenanceService: 20, epistemic-runtime: 45, plus existing engine tests).

**Impact:** The extensive test coverage was enabled by the TDD approach (Plans 02, 03, 05) and the established mock engine pattern from Phase 5. Test quality was high enough that the UAT found only pre-existing type errors, not behavioral bugs.

**Source:** 06.2-05-SUMMARY.md (test_results: 181/181), 06.2-UAT.md (Test checking: 104 tests, 10 files)

---

### UAT caught 9 pre-existing type errors that had survived previous typechecks
The UAT (06.2-UAT.md) found 9 pre-existing type errors across 4 files that had not been caught by earlier typechecks. These included PolicyEngine.ts missing key_index and PolicyHint fields, PolicyStoreFile.ts/test.ts missing key_index defaults, computeLayerStability bogus Effect requirements, and MaintenanceService.test.ts Effect.Ref.Ref vs Ref.Ref naming issues.

**Impact:** The type errors were introduced by earlier plans (06.2-01 extended PolicyEngineState with key_index, but PolicyEngine.ts and PolicyStoreFile.ts were not updated). The UAT's comprehensive typecheck caught what per-plan typechecks missed because per-plan typecheck scopes were too narrow.

**Source:** 06.2-UAT.md (Type checking section), 06.2-GAP-01-SUMMARY.md (Pre-existing Type Errors)

---

### Concept and causal parallel execution via Effect.all was straightforward
The parallel execution of concept.discover() and causalEngine.discover() via `Effect.all({ concept: ..., causal: ... }, { concurrency: 2 })` required no special infrastructure or testing -- Effect-TS handles the concurrency scheduling internally.

**Impact:** The Rust rayon::join pattern mapped cleanly to Effect.all with concurrency. The parallel execution required no changes to either engine's implementation -- they are pure Effect operations that are independently safe to run concurrently.

**Source:** 06.2-06-PLAN.md Task 2 (runDiscoveryPhases), 06.2-VERIFICATION.md Truth 18

---

### Verification gaps existed despite all tests passing
The verification pass found two gaps (local types in Surface.ts instead of contract imports, Aura.runMaintenance as Effect.die stub) even though all 104 tests were passing. The tests were testing the implementation against its own interfaces, not verifying cross-package wiring.

**Impact:** Tests ensure behavioral correctness but do not verify architectural constraints (import sources, layers). Dedicated key_link verification in the verification report is necessary to catch wiring gaps. The gap closure plans (GAP-01, GAP-02) fixed these without changing any test outcomes.

**Source:** 06.2-VERIFICATION.md (Key Link Verification section), 06.2-GAP-01-PLAN.md, 06.2-GAP-02-PLAN.md

---

### Surface functions required adapter bridges between contract and engine types
The getSurfacedPolicyHints implementation in EpistemicRuntime needed an adapter object to bridge between the contract PolicyHint type and the surface module's expected PolicyEngine interface. The adapter converts fields, maps states, and provides a `hints` property with a `values()` method that the surface function expects.

**Impact:** The adapter pattern adds a thin conversion layer but avoids modifying either the contract types or the surface module's interface. This is a consequence of D-11 (surface functions in engine packages) combined with D-02 (EpistemicRuntime as pure inspection surface) -- the two are structurally decoupled and need a bridge.

**Source:** 06.2-05-SUMMARY.md (Implementation notes - Policy surface adapter)

---

### Effect.Ref.Ref vs Ref.Ref naming ambiguity caused type errors
In MaintenanceService.test.ts, the type annotation `Effect.Ref.Ref<number>` caused TypeScript errors because the correct import syntax is `Ref.Ref<number>` (import Ref from effect, then use Ref.Ref). The UAT caught this pre-existing error and fixed it.

**Impact:** Effect-TS 4.0.0-beta.68 has a non-intuitive naming scheme for Ref types. The `Effect.Ref.Ref` pattern is incorrect -- it should be `Ref.Ref` after importing `import { Ref } from "effect"`. This is a version-specific quirk that will resolve in later effect versions.

**Source:** 06.2-UAT.md (Type checking section), 06.2-GAP-01-SUMMARY.md (Pre-existing Type Errors)
