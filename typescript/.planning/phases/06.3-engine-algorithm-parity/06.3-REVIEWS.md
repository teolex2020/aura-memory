# Cross-AI Plan Review — Phase 06.3

## Codex Review (GPT-5.4-high)

### Overall Assessment

The phase plan is well-sequenced at the engine level and shows strong awareness of the actual Rust parity gaps. The biggest strengths are its explicit artifact lists, dependency chain, and focus on deterministic behavior. The biggest weaknesses are cross-plan: contract-layer abstraction violations, too much reliance on hand-transcribed constants/tables instead of source-derived fixtures, and a validation strategy that waits too long to prove Rust parity. As written, the plans are directionally correct, but the phase still carries **high execution risk** until those issues are tightened.

### Cross-Plan Findings

#### Strengths
- The Belief → Concept → Causal → Policy order is sensible and matches dependency flow.
- The plans explicitly target the known audit gaps instead of drifting into unrelated work.
- Most plans include concrete acceptance criteria and intended tests.
- Determinism is treated as a first-class requirement.

#### Concerns
- **HIGH:** Plan 01 changes contract signatures but only typechecks `packages/contract`. That will break downstream packages immediately.
- **HIGH:** Several plans use concrete `*Impl` types inside contract/signature definitions. That conflicts with the project's effect-smol layering and contract purity rules.
- **HIGH:** Rust parity validation is deferred until the end, despite the phase goal being Rust algorithm parity. Unit tests alone are not enough here.
- **HIGH:** Many constants, maps, keyword lists, and stemming/equivalence tables look hand-curated from notes instead of source-extracted from Rust. That is a major transcription-risk.
- **MEDIUM:** Some plans use string unions for Rust-like enums, but the project requires TS string enums for runtime parity and testability.
- **MEDIUM:** Several later plans depend on fields not clearly added earlier, especially provenance arrays needed by Causal and Policy.

#### Suggestions
- Add repo-wide `bun run typecheck` gates after every contract/signature-changing plan.
- Replace concrete `*Impl` references in contracts with `*.Interface` references.
- Add per-engine Rust parity checkpoints after Belief, Concept, Causal, and Policy, not only at the end.
- Extract tables/constants from Rust or fixture them, instead of manually rewriting them from prose.
- Audit every new enum-like type against the project rule: use TS string enums, not ad hoc unions.

#### Overall Risk: **HIGH**
The plan is strong in intent, but the current validation and abstraction choices can let non-parity drift accumulate for a long time before it is detected.

### Per-Plan Review

#### 06.3-01 — Contract Type Expansions

**Strengths:**
- Front-loads schema/signature work that downstream plans need.
- Explicitly expands types to support richer Rust-aligned state.
- Correctly recognizes that Causal and Policy need more inputs.

**Concerns:**
- **HIGH:** `packages/contract`-only typecheck is insufficient after signature changes.
- **HIGH:** Contract imports concrete impl types rather than interface abstractions.
- **MEDIUM:** New enum-like values appear to be modeled as string unions in places, conflicting with project rules.
- **MEDIUM:** Some later-required fields are not clearly included yet, especially provenance fields for Policy suppression/seed selection.

**Suggestions:**
- Require workspace typecheck after this plan.
- Use `BeliefEngine.Interface`, `CausalEngine.Interface`, `ConceptEngine.Interface`, `PolicyEngine.Interface`.
- Make all Rust-enum equivalents TS string enums.
- Add missing provenance/report types now if later plans depend on them.

**Risk: HIGH** — Critical foundation plan; mistakes here will multiply downstream.

#### 06.3-02 — BeliefEngine P0

**Strengths:**
- Correctly targets the highest-impact Belief deviations first.
- Good focus on deterministic IDs, contradiction split, and incremental updates.
- Acceptance criteria are concrete.

**Concerns:**
- **HIGH:** `xxhash-wasm` `h64` is not Rust `xxh3_64`; deterministic is not the same as parity.
- **HIGH:** `update_with_sdr` is rewritten before subcluster parity is in place, so the structure may diverge from Rust and be reworked again in 06.3-03.
- **MEDIUM:** Content filter behavior is inconsistent between test wording and action wording.
- **MEDIUM:** Verification is package-local only.

**Suggestions:**
- Either implement true `xxh3_64` parity now or explicitly defer with a tracked non-parity marker.
- Keep `update_with_sdr` structurally aligned to Rust's coarse-group → subcluster → contradiction flow.
- Pin the exact byte-length rule from Rust and keep tests consistent with it.
- Add a small Rust fixture for hypothesis ID and contradiction split.

**Risk: MEDIUM-HIGH** — Good direction, but hashing and structural drift are serious parity risks.

## Final Recommendation

The plans are strong enough to proceed after revision, but I would not execute them unchanged. The highest-priority fixes are:

1. Replace contract references to concrete impl types with interface abstractions.
2. Add repo-wide typecheck gates immediately after signature/type changes.
3. Add per-engine Rust parity checkpoints before moving to the next engine.
4. Source-lock all constants, tables, and thresholds to Rust instead of prose summaries.
5. Resolve missing provenance fields before Causal/Policy implementation begins.

With those changes, the phase risk drops from **HIGH** to **MEDIUM**.
