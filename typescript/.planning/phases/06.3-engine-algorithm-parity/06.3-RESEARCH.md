# Phase 06.3: Engine Algorithm Parity - Research

**Researched:** 2026-05-28
**Domain:** Cross-language algorithm alignment (TypeScript ↔ Rust)
**Confidence:** HIGH

## Summary

This phase aligns the TypeScript BeliefEngine/ConceptEngine/CausalEngine/PolicyEngine with their Rust reference implementations, fixing 14 systematic deviations identified in AUDIT-DIFF.md. The primary recommendation is a phased line-by-line translation following the Rust source order (Belief → Concept → Causal → Policy), with each engine independently verified before proceeding.

**Key findings:**
- **BeliefEngine requires an architectural rewrite:** The TS hypothesis generation must change from Union-Find SDR clustering to Rust's contradiction-based split (`split_by_contradiction`). This changes the number of hypotheses per belief from N (SDR clusters) to max 2 (supporting/opposing).
- **CausalEngine requires the largest change:** The entire signal source concept must change from concept co-occurrence to record-level edge extraction (explicit caused_by_id + connection_type causal + temporal edges). This means the engine signature MUST change from accepting `ConceptEngineState` to accepting `BeliefEngine.Interface`.
- **PolicyEngine signature MUST change:** `discover()` must accept three engines (causal + concept + belief) instead of just `causal_state`. This is a contract change that affects the MaintenanceService orchestration.
- **ConceptEngine changes are additive:** CanonicalFeature mode and cluster guards extend existing functionality without requiring contract changes.
- **Existing tests will break:** Almost all existing test fixtures will need updating because hypothesis structures, IDs, and scoring will change significantly.

**Primary recommendation:** Execute per D-01 (Belief → Concept → Causal → Policy) with D-04 hybrid strategy (P0 rewrites from Rust source, P1/P2 as checklist fixups).

## Critical Contract Issue

**Finding:** D-03 (contract decides) states engine interfaces will not change, but both CausalEngine and PolicyEngine `discover()` signatures MUST change to accept the inputs required by Rust algorithms.

- **CausalEngineImpl.discover** currently accepts `(concept_state: ConceptEngineState, ...)` — Rust needs `(belief_engine: &BeliefEngine, records, sdr)`. The algorithm requires knowledge of belief-level state (via `record_index → hypothesis → belief`) to aggregate edges to belief-level patterns.
- **PolicyEngineImpl.discover** currently accepts `(causal_state: CausalEngineState, records)` — Rust needs `(causal_engine, concept_engine, belief_engine, records)`. The algorithm requires all three engines for polarity classification, concept lookup, and belief gate.

**Recommendation:** Update the interface type definitions in `@aura/contract` (Causal.ts, Policy.ts) and the MaintenanceService orchestration calls. This is unavoidable — the Rust algorithms cannot be faithfully mirrored without these changes.

### Patterns

**Pattern 1: Rust 1:1 Translation Pattern**

The core approach for each engine function. The Rust source is the authoritative spec; the TS version should mirror it function-by-function with the same control flow, constants, and variable names in camelCase.

```typescript
  const self = this;
  return Effect.gen(function* () {
    const now = yield* Clock.nowSeconds();

      const key = yield* self.claim_key_with_mode(/* ... */);
      // ...
    }
    // Step 2: For each coarse group, apply subcluster strategy
    // Step 3: Create/update beliefs via split_by_contradiction
    // Step 4: Resolve beliefs
    // Step 5: Prune stale beliefs
  });
}
```

**Pattern 2: Subcluster Strategy Dispatch**

```typescript
// Rust: match coarse_key_mode { ... } → picks subcluster fn + threshold
const threshold = /* mode-dependent threshold mapping */;

```

### Anti-Patterns to Avoid

- **Using `id12()` for hypothesis IDs**: Breaks deterministic stability tracking. Must use `xxh3_64(belief_id + "\0" + sorted_record_ids)` (or xxh64 approximation).
- **Removing beliefs on deprecation**: Rust halves confidence and sets state to Unresolved — it does NOT delete the belief. The TS `deprecate_belief` must change.
- **Full state rebuild**: The TS `update_with_sdr` currently clears state and rebuilds. Must become incremental (look up by key_index, reuse existing beliefs when possible).
- **Concept-level causal patterns**: The TS CausalEngine operates on concept co-occurrence. Must switch to record-level edge extraction then aggregate to belief-level patterns.

## Common Pitfalls

### Pitfall 1: Hypothesis Score Formula Difference (Variance: Sample vs Population)
**What goes wrong:** Rust uses sample variance `/(n-1)` but TS uses population variance `/n`.
**Why it happens:** Rust `consistency` computation uses `(confidences.len() - 1)` for variance divisor; TS uses `confidences.length`.
**How to avoid:** Change `computeConsistency` to divide by `confidences.length - 1` (sample variance). Handle `length === 1` case returning 1.0.
**Warning signs:** Hypothesis scores off by small constant when hypothesis has >1 records.

### Pitfall 2: Content Filter — Token Count vs Byte Length
**What goes wrong:** Rust filters on `content.len() < 10` (byte length); TS uses `estimateTokens() < 5` (token estimate).
**Why it happens:** Rust checks raw byte length; TS estimates word/token count via Segmenter.
**How to avoid:** Replace `estimateTokens(rec.content) < MIN_CONTENT_TOKENS` with `rec.content.length < 10` (or `rec.content.length < 10 && rec.content.trim().length < 10`).
**Warning signs:** Different set of records entering belief formation.

### Pitfall 3: CoarseKey Standard Mode — Tag Truncation
**What goes wrong:** Rust `Standard` mode truncates to top-3 sorted tags; TS uses ALL sorted tags.
**Why it happens:** Rust `tags.truncate(3)` after sorting; TS `activeTags.join(",")` without truncation.
**How to avoid:** Add `tags.truncate(3)` (or `activeTags.slice(0, 3)`) in `Standard` mode's tag processing.
**Warning signs:** Different coarse group keys between Rust and TS for records with 4+ tags.

### Pitfall 4: TagFamily Mode — Alphabetically First vs Prefix Extraction
**What goes wrong:** Rust uses `tags.sort(); tags.first()` (alphabetically first tag); TS extracts prefix before `/` or `_`.
**Why it happens:** TS implements its own "family extraction" logic; Rust keeps the raw tag as-is.
**How to avoid:** Rewrite `TagFamily` mode to match Rust: take the alphabetically first tag as the sole family tag.
**Warning signs:** Different coarse group keys for records where alphabetic first tag differs from prefix.

### Pitfall 5: CoarseKey DualKey/NeighborhoodPool — Uses Broad Corridor Only
**What goes wrong:** Rust `DualKey` uses `namespace:semantic_type` (no tags in key) with `tag_guarded` subcluster; TS currently uses `namespace:all_tags:semantic_type`.
**Why it happens:** TS doesn't have the DualKey-separate implementation.
**How to avoid:** Change `DualKey` and `NeighborhoodPool` to produce `namespace:semantic_type` as the coarse key, and dispatch to `sdr_subcluster_tag_guarded` for fine grouping.
**Warning signs:** Different coarse group counts.

### Pitfall 6: CausalEngine Interface Signature Change
**What goes wrong:** Current `CausalEngineImpl.discover` signature accepts `ConceptEngineState` but Rust requires `BeliefEngine`.
**Why it happens:** Rust builds `record_index → hypothesis → belief` mapping for aggregation; TS doesn't have access to this.
**How to avoid:** Change the contract `Causal.ts` interface type. Also update `CausalEngineState` to include the richer pattern structure (20+ fields).
**Warning signs:** n/a (compile-time type error if interface changes).

### Pitfall 7: PolicyEngine Interface Signature Change
**What goes wrong:** Current `PolicyEngineImpl.discover` signature accepts `(CausalEngineState, records)` but Rust requires `(CausalEngine, ConceptEngine, BeliefEngine, records)`.
**Why it happens:** Rust needs concept lookup (`build_belief_to_concepts`) and belief confidence aggregation (`aggregate_belief_confidence`) from the other two engines.
**How to avoid:** Change the contract `Policy.ts` interface type. The function signature should become `discover(causal_engine, concept_engine, belief_engine, records)`.
**Warning signs:** n/a (compile-time type error if interface changes).

### Pitfall 8: apply_layer_feedback — Complete Mismatch
**What goes wrong:** TS `apply_layer_feedback` accepts `{ belief_id, action, factor }[]`; Rust accepts `&CausalEngine, &PolicyEngine` and processes signal from both.
**Why it happens:** The Rust implementation cross-references policy hints (action_kind, policy_strength) and causal patterns (causal_strength, evidence counters) to compute confidence_delta and volatility_delta per belief_id.
**How to avoid:** Rewrite `apply_layer_feedback` to accept CausalEngine + PolicyEngine types and iterate their internal state.
**Warning signs:** Current TS implementation returns `{ applied: number }` but Rust returns `BeliefFeedbackReport` with 6 fields.

### Pitfall 9: Evidence Gates — Support Gate Threshold
**What goes wrong:** Rust's `meets_support_gate` checks `support_count >= 2` (or `explicit_support_count >= 1` in ExplicitTrusted mode); TS has no evidence gates.
**Why it happens:** TS state classification uses simple `confidence > 0.7 → Stable`, bypassing evidence quality checks.
**How to avoid:** Implement `meets_support_gate`, `meets_repeated_evidence_gate`, `meets_counterfactual_gate`, and require all three for Candidate/Stable promotion.
**Warning signs:** TS promotes patterns to Stable/Singleton that Rust would reject.

### Pitfall 10: Concept SimilarityMode — CanonicalFeature vs SdrTanimoto
**What goes wrong:** Rust defaults to `CanonicalFeature` (Jaccard over canonical tokens); TS only implements `SdrTanimoto`.
**Why it happens:** TS `ConceptSimilarityMode` enum has both values, but the engine only switches on `SdrTanimoto`.
**How to avoid:** Implement the `CanonicalFeature` branch: stem words → apply equivalence dict → compute Jaccard over canonical token sets. Default to `CanonicalFeature` matching Rust.
**Warning signs:** Concept clusters differ between Rust and TS for short-text records.

```typescript
// Rust: xxh3_64(belief_id + "\0" + sorted_record_ids) → 12-char hex
function deterministicHypothesisId(
  hasher: { h64: (input: string) => bigint },
  belief_id: string,
  records: ReadonlyArray<AuraRecord>
): string {
  const ids = [...records].map(r => r.id).sort();
  const buf = [belief_id, ...ids].join('\0');
  const hash = hasher.h64(buf) & ((1n << 64n) - 1n);
  return hash.toString(16).padStart(12, '0');
}
```

```typescript
function splitByContradiction(

  const supporting: AuraRecord[] = [];
```

```typescript
  records: ReadonlyArray<AuraRecord>,
  sdrLookup: SdrLookup,
  threshold: number
): AuraRecord[][] {
  const n = records.length;
  if (n <= 1) return records.map(r => [r]);
  const parent = new Array<number>(n).fill(0).map((_, i) => i);
  // union-find find + union helpers...
  for (let i = 0; i < n; i++) {
    const sdrI = sdrLookup.get(records[i].id);
    if (!sdrI) continue;
    const tagsI = new Set(records[i].tags);
    for (let j = i + 1; j < n; j++) {
      // Tag barrier: require shared tags >= 1 (unless one side has no tags)

      if (!sdrJ) continue;
      if (tanimoto(sdrI, sdrJ) >= threshold) union(parent, i, j);
    }
```

### Policy: Polarity classification

```typescript
// Rust source: policy.rs:423-496
function classifyPolarity(
  effectRecordIds: string[],
  records: Map<string, AuraRecord>
): 'Positive' | 'Negative' | 'Neutral' {
  let positiveSignals = 0;
  let negativeSignals = 0;
  for (const eid of effectRecordIds) {
    const rec = records.get(eid);
    if (!rec) continue;
    // semantic_type == "contradiction" → -2
    if (rec.semantic_type === "contradiction") negativeSignals += 2;
    // tags match keywords
    for (const tag of rec.tags) {
      if (NEGATIVE_KEYWORDS.some(kw => tag.toLowerCase().includes(kw))) negativeSignals += 1;
      if (POSITIVE_KEYWORDS.some(kw => tag.toLowerCase().includes(kw))) positiveSignals += 1;
    }
    // content match keywords
    const lower = rec.content.toLowerCase();
    for (const kw of NEGATIVE_KEYWORDS) { if (lower.includes(kw)) negativeSignals += 1; }
    for (const kw of POSITIVE_KEYWORDS) { if (lower.includes(kw)) positiveSignals += 1; }
  }
  if (negativeSignals > positiveSignals && negativeSignals >= 2) return 'Negative';
  if (positiveSignals > negativeSignals && positiveSignals >= 2) return 'Positive';
  return 'Neutral';
}
```
