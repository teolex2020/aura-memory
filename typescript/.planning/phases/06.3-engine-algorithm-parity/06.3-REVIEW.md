## Warnings

### WR-01: BrainAuraRecord structurally incompatible with AuraRecord in runMaintenance

**File:** `packages/core/src/Aura.ts:410`
**Issue:** `Aura.runMaintenance()` casts `this.records` (type `BrainAuraRecord[]`, loaded from `brain.aura`) to `AuraRecord` via `r as unknown as AuraRecord`. `BrainAuraRecord` (defined in `packages/storage/src/BrainAura.ts:10-22`) has completely different fields from `AuraRecord` (contract `Record`): `BrainAuraRecord` has `text`, `dna`, `timestamp`, `sdr_indices` etc., while `AuraRecord` has `content`, `namespace`, `semantic_type`, `tags`, `created_at` etc. Engines receiving these cast records will encounter `undefined` for `content`, `namespace`, etc., causing all records to be silently skipped by the content length filter (`byteLength(undefined)` produces 9 which is < 10). The `store()`, `update()`, and `connect()` methods correctly use `loadCognitiveRecords()` which returns proper `AuraRecord` via `normalizeCognitiveRecord()`, but `runMaintenance()` bypasses that path entirely.

**Fix:** Either convert `BrainAuraRecord` to `AuraRecord` explicitly before passing to engines, or use `loadCognitiveRecords(dir)` in `runMaintenance()` instead of `this.records`. The latter aligns with all other methods on Aura:
```typescript
// In runMaintenance, replace:
const recordsList = this.records
// ...
const records = new Map<string, AuraRecord>(
  recordsList.map((r) => [r.id, r as unknown as AuraRecord])
)
// With:
const records = yield* loadCognitiveRecords(dir)
```

### WR-02: generateRecommendation uses raw string literals instead of PolicyActionKind enum

**File:** `packages/policy/src/PolicyEngine.ts:519-532`
**Issue:** The `generateRecommendation` function uses raw string literals (`"avoid"`, `"verify_first"`, `"prefer"`, `"recommend"`, `"warn"`) in its `switch` cases instead of TypeScript string enum values (`PolicyActionKind.Avoid`, `PolicyActionKind.VerifyFirst`, etc.). While behavior is identical at runtime (since `PolicyActionKind.Avoid = "avoid"`), the project convention (as fixed in CR-03 for `PolicyState`) requires enum value references to ensure compiler-verified correctness if enum values change.

**Fix:**
```typescript
switch (actionKind) {
  case PolicyActionKind.Avoid:
    return `Avoid: '${causeSummary}' in domain [${domain}] has been associated with negative outcomes.`
  case PolicyActionKind.VerifyFirst:
    return `Verify first: '${causeSummary}' in domain [${domain}] has shown risk signals — check before proceeding.`
  case PolicyActionKind.Prefer:
    return `Prefer: '${causeSummary}' in domain [${domain}] has consistently led to positive outcomes.`
  case PolicyActionKind.Recommend:
    return `Recommend: '${causeSummary}' in domain [${domain}] has shown positive signals.`
  case PolicyActionKind.Warn:
    return `Warning: '${causeSummary}' in domain [${domain}] has a strong causal pattern but unclear polarity.`
  default:
    return `Warning: '${causeSummary}' in domain [${domain}] has a strong causal pattern but unclear polarity.`
}
```

### WR-04: Policy Surface.ts uses deprecated local interfaces incompatible with contract

**Files:** `packages/policy/src/Surface.ts:46-76, 134, 147`
**Issue:** The policy surface functions (`surfacePolicyHints`, `surfacePolicyHintsFiltered`) accept the deprecated local `PolicyEngine` interface (line 73-76) which has `hints: ReadonlyMap<string, PolicyHint>` and `keyIndex: ReadonlyMap<string, string>` -- a flat data container. This is NOT the contract's `PolicyEngine.Interface` (which exposes `discover()`, `retract_hint()`, `stats()` methods). The deprecated local `PolicyHint` (lines 46-64) also uses camelCase fields (`triggerCausalIds`, `supportingRecordIds`) that differ from the contract's snake_case `PolicyHint` type (`cause_key`, `effect_keys`, `cause_record_ids`). This means the surface functions cannot be called with a real `PolicyEngine.Interface` without an adapter layer that translates the engine's internal hints into this local shape. The `@deprecated` JSDoc annotations are present (lines 38-44, 69-70) but the surface functions still depend on the deprecated types.

**Fix:** Either:
1. Make the surface functions accept `PolicyEngine.Interface` (via `engine.stats()`) and use the contract's `PolicyHint` type directly, or
2. Add an explicit adapter function `fromEngineState(state: PolicyEngineState): PolicyEngine` and document the conversion.

### WR-05: computeCorpusFingerprint hash drift -- might differ between first and subsequent cycles

**File:** `packages/causal/src/CausalEngine.ts:863-869`
**Issue:** `computeCorpusFingerprint()` preferentially uses `_hasher` (xxhash-wasm) if available, falling back to `simpleHash64`. The `_hasher` is initialized asynchronously via `getHasher()` (line 917-920). In `CausalEngineImpl.discover()`, `computeCorpusFingerprint` is called synchronously at the start (line 956) before `aggregateToPatterns` (which is the only code path that calls `getHasher()`). On the first maintenance cycle, `_hasher` is `null`, so `simpleHash64` is used to produce the fingerprint and it is stored in `last_corpus_fingerprint`. On subsequent cycles, if `_hasher` has been initialized, a DIFFERENT fingerprint is produced by xxhash for the same corpus, causing the skip optimization to be missed and defeating its purpose.

**Fix:** Eager-initialize the hasher before fingerprint computation:
```typescript
// In CausalEngineImpl.discover(), before fingerprint check:
if (!_hasher) {
  _hasher = await getHasher()
}
const fingerprint = computeCorpusFingerprint(_records)
```

### IN-04: dead code in EpistemicRuntime.test.ts (sampleBelief's hypothesis_ids set to empty)

### TODO-W02: 6 UNIMPLEMENTED methods use Effect.die() — hard crash surface for callers (WARNING)

**File:** `packages/core/src/Aura.ts:353, 362, 537, 546, 555, 564`
**Issue:** Six public methods throw `Effect.die(new UnimplementedError(...))`:
| Line | Method |
|------|--------|
| 353 | `explain_recall()` |
| 362 | `explain_record()` |
| 537 | `get_relation_graph()` |
| 546 | `get_entity_graph()` |
| 555 | `get_project_graph()` |
| 564 | `get_family_graph()` |

`Effect.die()` is an unrecoverable fiber death — callers cannot catch it with `Effect.catchAll`. These should use `Effect.fail(new UnimplementedError(...))` so callers can handle the "not implemented" case gracefully. The current behavior will crash any fiber that calls these methods, even in a production maintenance cycle that doesn't need the graph APIs.

**Fix:** Replace `Effect.die(new UnimplementedError(...))` with `Effect.fail(new UnimplementedError(...))`, or update `Aura.Interface` to mark these methods as optional/defect-bearing.
